# v0.3 Acceptance: Host Conformance

v0.3 exists to prove that Memory Engine is a reusable library, not a Telegram bot with a strong private core.

The owner should not be the test harness. Manual Telegram/Godot clicking is allowed only as final smoke testing. The primary acceptance path is automated host conformance: every host must pass the same memory scenario through a small driver interface and prove the same engine invariants.

## Goal

By the end of v0.3, the project must support:

- Telegram bot host;
- Godot host;
- direct/local host used as the deterministic baseline.

Accepted v0.3 scope is direct/local, Telegram bot through `telegram-local`, and
Godot through `godot-headless`. Telegram userbot is explicitly out of v0.3 scope
until a real userbot adapter is chosen; it must pass the same contract before it
can be claimed later.

Each host may have different transport and UI code, but none of them may own memory policy. A host is allowed to:

- receive a user text event;
- call Memory Engine;
- render/load prompts outside the core;
- execute `LlmRequest -> text` with its provider;
- submit the result back to Memory Engine;
- show replies, debug output, and telemetry.

A host is not allowed to decide:

- what counts as memory;
- what goes to Core;
- which memory unit is valid;
- which candidate is promoted;
- what should be forgotten;
- how prompt-facing memory is structured.

Those decisions belong to the Rust core.

## Host Driver Contract

Every conformance driver exposes this conceptual interface:

```text
send_user_message(text) -> assistant_reply
run_sleep() -> sleep_report
run_reflection() -> candidate_report
inspect_core() -> core_snapshot
inspect_archives() -> archive_snapshot
inspect_prompt_view(current_user_message) -> rendered_memory_view
```

The direct driver may fake the LLM responses. Live drivers may call a real model, but the test must still assert engine state rather than judge prose by feel.

## Acceptance Matrix

| Layer | Direct/local | Telegram bot | Telegram userbot | Godot |
|---|---:|---:|---:|---:|
| Core replay / deterministic fake LLM | required | not applicable | not applicable | not applicable |
| Host conformance scenario | required | required | out of scope | required |
| Transport smoke | not applicable | covered by telegram-local host path | out of scope | required |
| Live LLM smoke | optional | deferred smoke, not v0.3 gate | out of scope | optional |
| Manual owner check | not required | optional | optional | optional |

## Required Scenario

The shared scenario must include at least:

1. a stable self-statement about the user's name;
2. a stable personal memory about the user's cat Irzha;
3. one ordinary topic, such as space or aircraft;
4. a sleep cycle;
5. a context package and rendered memory view check.

The driver must assert:

- sleep completes and creates a Complete archive;
- memory units are created;
- Core receives qualified stable facts through the core-owned bridge;
- prompt view contains separate `core_memory`, `long_memory`, `short_memory`, and `current_user_message` sections;
- the current user message is not duplicated into short memory;
- host code did not need to inspect internal memory tracks such as `personal_signals` to render the prompt.

## First Implementation Slice

The first v0.3 slice is `tests/host_conformance/host_conformance.py` with `--host direct`.

This driver uses the Python adapter and deterministic fake LLM responses. It proves the host-facing JSON/PyO3 surface can run the same scenario without Telegram, Godot, or manual clicking.

The second slice is the same runner with `--host telegram-local`.

This driver imports the Telegram host, uses fake Telegram transport and fake Gemini, and proves the Telegram host code path can pass the same scenario without the real Telegram API.

The third slice starts the Godot path:

- `crates/godot_adapter/` exposes a thin Godot GDExtension class with the same JSON-string boundary as the Python adapter;
- `hosts/godot_headless/` contains a minimal headless Godot project and GDScript conformance runner;
- `tests/host_conformance/host_conformance.py --host godot-headless` builds the extension, copies it into a temporary Godot project, and runs the shared scenario if a Godot executable is available through `GODOT_BIN`, `--godot-bin`, `godot4`, or `godot`.

This slice was accepted on 2026-06-10 with Godot 4.6 stable console. The adapter targets Godot 4.6 through godot-rust `0.5.3`; older Godot runtimes are not accepted until the adapter is explicitly rebuilt against an older API feature and re-tested.

Deferred slices:

- a Telegram-live smoke driver for one real bot run, as transport smoke only;
- a CI job or documented pinned local toolchain for Godot 4.6 headless conformance.

These are useful maintenance tasks, but they are not v0.3 closure gates because
the accepted v0.3 hosts already pass automated conformance without making the
owner the test harness.

## Non-Goals

v0.3 does not claim:

- vector recall;
- better memory quality benchmarks;
- a polished Godot UI;
- multiple Telegram accounts running in production;
- a public benchmark result.

Those can come later. v0.3 is about reusable host boundaries.

## Gates

v0.3 closes when the accepted hosts pass:

- `cargo test --workspace`;
- `cargo clippy --workspace --all-targets -- -D warnings`;
- `python -m pytest crates/python_adapter/tests -q`;
- `python tests/host_conformance/host_conformance.py --host direct`;
- `python tests/host_conformance/host_conformance.py --host telegram-local`;
- `python tests/host_conformance/host_conformance.py --host godot-headless` with `GODOT_BIN` or `--godot-bin`;
- equivalent conformance command for every host accepted into v0.3 scope.

## Close Status

Closed on 2026-06-10.

Accepted hosts:

- direct/local: passed through `tests/host_conformance/host_conformance.py --host direct`;
- Telegram bot host path: passed through `--host telegram-local` with fake Telegram and fake Gemini, exercising real `bot.py`;
- Godot: passed through `--host godot-headless` with real Godot 4.6 stable console, `memory_units=3`, `core_facts=3`.

What v0.3 proves:

- Memory Engine is reusable through thin host adapters.
- Direct, Telegram-local, and Godot-headless run the same scenario.
- Memory policy remains in the Rust core; hosts dispatch, render/load prompts,
  execute model requests, and display/telemetry results.

What v0.3 does not claim:

- polished Godot/Chibigochi gameplay integration;
- Telegram userbot support;
- vector recall;
- benchmarked memory quality;
- multi-host production deployment.
