# Архітектура Memory Engine v0.2

## Для чого існує цей документ

Стратегія в `docs/strategy.md` пояснює задум: окрема Rust-бібліотека тришарової пам'яті, яка обслуговує кілька різних застосунків через тонкі адаптери. Архітектурний документ робить наступний крок. Він не дає Rust-коду і не дає точних JSON-схем. Він фіксує **межі**, на яких ми домовляємось перед першим рядком коду: які поняття існують, як вони називаються, як ядро системи спілкується з LLM, як працює recall, як влаштоване сховище, що вже увійшло у v0.1/v0.2 і що чесно відкладено на v0.3+.

Документ написаний так, щоб людина або ШІ, які прочитали стратегію і відкривають його вперше, могли зрозуміти задум без жодного зовнішнього контексту. Він є основою для двох наступних артефактів: контрактів даних (точні JSON-схеми) і архітектури Rust-модулів. Без нього ці артефакти писати рано.

Версія документа v0.2, синхронізована 2026-06-10 після post-v0.2 audit. Будь-яка зміна архітектурних рішень нижче має проходити через `HISTORY.md`.

## 1. Термінологія

Стратегія використовує слово "core" у двох сенсах, і це збиває. Архітектура жорстко розводить терміни. Усі подальші документи і код мають триматись цих імен.

**ядро системи** — уся Rust-бібліотека Memory Engine як одне ціле. Одна crate, один артефакт релізу. Містить логіку шарів, recall, sleep, reflection, scoring, decay, storage, версіонування і контракт `LlmAdapter`. Ядро системи синхронне і не виконує жодного мережевого I/O.

**шар Ядро** — стабільний шар пам'яті, один із трьох (поруч із шарами Сесія і Архів). Зберігає довготривалі факти, синтезовані переконання, сталі висновки про користувача чи персонажа. Змінюється повільно і за явними правилами промоції.

**шар Сесія** — робоча пам'ять поточної взаємодії: живий день у грі, поточна розмова в боті, поточна робоча сесія в третьому проєкті. Має ліміт розміру і живий хвіст для LLM-контексту.

**шар Архів** — довгострокова пам'ять. Стиснуті спогади, які пройшли відбір під час sleep. Структурована, індексована, версіонована.

**хост** (або **хост-проєкт**) — застосунок, що підключає ядро системи: Chibigochi, Telegram-бот, або інший майбутній продукт.

**адаптер** — тонкий шар коду між хостом і ядром системи. Перекладає події конкретного домену у контракти ядра і повертає результат у форматі, зручному хосту. Для Python — обгортка через PyO3. Для Godot — GDExtension. Адаптер живе у репозиторії ядра системи як окремий sub-crate або біля нього.

**подія** (Event) — нормалізоване повідомлення хоста про те, що сталося у його світі. Єдиний канал, через який інформація заходить у ядро системи.

**PendingTask** — об'єкт, що описує мовну задачу, яку ядро системи делегує LLM через хост. Ядро повертає його замість того, щоб виконувати мережевий виклик саме. Хост виконує задачу і повертає результат через `resume()`.

**recall** — запит до ядра системи на релевантні спогади у заданому контексті. Повертає список елементів із ваги, джерела, часу, теми і пояснення релевантності.

**sleep** — процес консолідації шару Сесія у шар Архів. Запускається тригером від хоста (кінець розмови, сон героїні, завершення робочої сесії).

**reflection** — процес довшого осмислення накопиченого Архіву. Шукає повторювані теми, патерни, кандидатів на промоцію у шар Ядро.

**Core Store** — сама стабільна основа пам'яті всередині шару Ядро: довготривалі факти, переконання, сталі висновки. Належить ядру системи.

**Core Context Package** — готовий контекст для LLM, зібраний із Core Store разом із поточним доменним станом, що його приходить від хоста. Ядро системи допомагає його зібрати, але джерело поточного доменного стану залишається в хост-проєкті.

**candidate belief** — потенційний стабільний висновок, виявлений reflection-процесом. Чекає на промоцію в Core Store за явними правилами.

**memory unit** — атомарна одиниця спогаду: один змістовний епізод або висновок, який має власну коротку тезу, вагу, джерела і статус. Повні технічні ідентифікатори живуть у storage/debug; у prompt і agent-review можна використовувати короткі локальні мітки на кшталт `m1`, `m2`, щоб не витрачати токени на довгі ID.

**fidelity validator** — LLM-задача або агент-перевірка, яка не вирішує, що є "істинним" у світі, а перевіряє, чи стиснений спогад чесно відповідає raw-подіям, з яких він зроблений. Validator може позначити unit як `valid`, `too_broad`, `unsupported`, `distorted` або `missing_key_detail`, але не промоутить нічого в Core напряму.

**LlmAdapter** — Rust trait, який реалізує хост або зовнішній модуль. Дає ядру системи спосіб попросити мовну задачу, не знаючи про конкретного провайдера.

## 2. Загальна форма

Ядро системи живе на межі між подіями застосунку і довготривалою пам'яттю. Хост знає, що сталося у його світі. Ядро системи знає, як перетворити це на пам'ять.

Потік номер один іде **від хоста до ядра системи**: події, стани, повідомлення, тригери sleep і reflection. Хост-адаптер нормалізує їх у тип `Event` і викликає `engine.ingest(event)`. Подія потрапляє у шар Сесія.

Потік номер два іде **від ядра системи до хоста**: коли ядру треба мовна задача (стискання сесії, оцінка значущості, reflection-висновок), воно повертає `PendingTask`. Хост виконує задачу через свій LLM-стек і повертає результат через `engine.resume(task_id, result)`.

Потік номер три — **recall**. Хост викликає `engine.recall(query)` і отримує список релевантних спогадів. Це синхронний виклик, він не вимагає LLM (на v0.1).

Потік номер чотири — **sleep**. Хост викликає `engine.sleep(session_id)`. Ядро системи виконує алгоритмічну частину консолідації синхронно і повертає `PendingTask` для LLM-частини. Хост виконує задачу, повертає результат, ядро завершує sleep і повертає підсумок.

Ці чотири потоки — повний публічний контракт ядра системи. Решта (decay, перевірка цілісності, міграції) виконується всередині під час відповідних викликів.

## 3. Архітектурні рішення v0.1

Чотири рішення нижче є несучими. Вони формують усю решту контрактів. Кожне зафіксоване з обґрунтуванням, щоб майбутні зміни робились свідомо.

### 3.1 Як ядро системи спілкується з LLM: задачно-орієнтована модель

Ядро системи синхронне і **не виконує жодного мережевого I/O**. Коли логіка ядра потребує мовної задачі, воно повертає `PendingTask` із усіма даними, потрібними для виконання: тип задачі, вхід, ідентифікатор промпта, очікувана форма результату. Хост виконує задачу — синхронно, асинхронно, через чергу, через будь-якого провайдера — і повертає результат через `engine.resume(task_id, result)`.

Це рішення дає чотири властивості:

1. **Виживання при lifecycle-обриві.** Мобільний застосунок може бути вбитий системою посеред sleep. `PendingTask` серіалізується на диск як частина стану ядра. Після перезапуску ядро бачить незавершений task і знає, де продовжити.
2. **Структурна провайдер-агностичність.** Ядро системи фізично не може звернутись до OpenAI, Google, Anthropic чи іншого провайдера. Не "ми домовились не зашивати" — а "технічно не можемо". Це найсильніша форма межі.
3. **Тривіальне тестування без LLM.** Тест підставляє підготовлений результат через `resume()`. Жодних моків мережі.
4. **Сумісність із будь-яким хост-середовищем.** Godot не має async runtime — message passing йому природний. Python asyncio з message passing щасливий. Майбутній хост на іншій платформі також.

Ціна рішення: API ядра системи має явну точку входу `resume()` і явну модель стану "очікую на task". Контракт `LlmAdapter` стає більш формальним. Це адекватна ціна за роботу впродовж років.

### 3.2 Recall: гібридний пайплайн з трьох стадій

Recall у ядрі системи спроєктований як **трирівневий пайплайн** з самого початку. На v0.1 активна тільки перша стадія. Решта мають готовий контракт і вмикаються через конфіг тоді, коли потрібні.

**Стадія 1 — структурний фільтр і детермінована формула.** Працює завжди. Фільтрує кандидатів по тегах, темі, часовому вікну, порозі ваги, прив'язках до шару Ядро. Ранжує простою формулою: вага × decay × відповідність контексту запиту × бонус за зв'язок із поточною темою. Не потребує LLM, не потребує embeddings. Детермінована, швидка, пояснювана.

**Межа сесії у recall.** Якщо `RecallQuery.session_id` заданий, stage1 archive recall розглядає тільки `ArchiveEntry` з таким самим `source_session_id`. Це default для `core_context_package` і звичайного chat prompt: нова сесія не має підхоплювати старі archive memories тільки тому, що запит семантично схожий. Глобальний пошук по всьому Архіву дозволений лише для явного admin/debug сценарію, коли хост передає `session_id: null`.

**Стадія 2 — re-ranking за embeddings.** На v0.1 не реалізована, але формат архівного спогаду вже містить поле `embedding_model_id` і місце під вектор. Працює як re-ranking над топ-K кандидатів зі стадії 1, а не як основний пошук. Це принципово: ми не робимо vector search замість фільтрів, ми додаємо semantic re-rank поверх них. Embeddings рахуються в момент створення архівного спогаду через `PendingTask`. Поле `embedding_model_id` дозволяє відслідкувати, які записи embedovaní якою моделлю, і безпечно мігрувати при зміні моделі.

На практиці Stage 2 не входить у найближчу v0.2. Векторне сховище вмикається тільки після стабілізації Core/reflection/forgetting і тільки як opt-in feature. Частина користувачів не захоче зберігати семантичні вектори про себе; хост має мати явний перемикач "без embeddings" / "з embeddings". При вимкнених embeddings Stage 1 лишається повністю працездатним.

**Стадія 3 — LLM re-ranking.** На v0.1 не реалізована, але передбачена. Промпт типу "ось запит у такому контексті, ось N кандидатів, поверни тих, що дійсно релевантні, з поясненням". Виконується через `PendingTask`. Корисна там, де embeddings і фільтри пропускають семантично складні випадки.

Чому це правильно як стартова архітектура, а не "почати з простого, потім додати": pure-vector search провалюється на часових запитах і точних збігах. Pure-filter провалюється на парафразах. Серйозні системи пам'яті в індустрії конвертують до hybrid pipeline. Закладаючи його одразу як форму, ми не доплачуємо складність — ми не починаємо потім переписувати.

### 3.3 Сховище: файли як джерело правди, з абстракцією під індекс

Канонічне сховище ядра системи — **файли на диску**: JSON або JSONL для машинно-читабельних шарів і `session.md` як людський журнал поточної сесії. Archive/Core записи наразі не мають окремих Markdown-дампів: pretty-printed JSON є поточною human-inspectable формою для audit/debug, а `.md` exports відкладені до реального workflow, який їх потребуватиме. Розкладка по шарах і по сесіях, як зафіксовано в `memory/README.md`. Жодного прихованого формату.

Ядро системи має `Storage` як Rust trait. Дефолтна імплементація — файлова. Можливість індексованої імплементації (наприклад, sled чи sqlite, що тримає індекс над файлами-джерелами) **закладена в інтерфейс**, але не реалізована на v0.1. Якщо індекс додається, файли залишаються правдою, індекс — кешем, перебудовним з файлів.

Crash safety реалізується двома механізмами:

1. **Атомарний запис одного файлу:** запис у тимчасовий файл, fsync, перейменування. Стандартний UNIX-патерн.
2. **Журнал операцій для мульти-файлових змін:** у v0.2 це зарезервований storage primitive і цільовий механізм для майбутніх migration/recovery-heavy операцій. Runtime sleep зараз не пише transaction journal; crash safety забезпечується атомарним записом окремих файлів, durable `SleepRun` checkpoints і ідемпотентним coverage-cache.

Чому файли як правда: принцип "керованість для людини" зі стратегії — це обмеження, а не побажання. Людина має могти зайти у `memory/sessions/2026-05-17_001/session.md` і прочитати живий журнал сесії, а archive/core записи мають бути відкритими як pretty JSON, не схованими в opaque базі. SQLite-only це порушує. Файли як правда — єдиний варіант, що чесно дотримує принцип. Абстракція `Storage` страхує нас від тупика, коли архів виросте і знадобиться індекс.

### 3.4 Перший хост: Telegram-бот через PyO3

Перша інтеграція — Python-адаптер до Telegram-бота через PyO3. Godot-адаптер через GDExtension робиться другим, коли публічне API ядра системи вже відшліфоване ботом.

Це рішення про **порядок реалізації**, а не про спеціалізацію. Контракти в цьому документі і в наступних задаються одразу з прикладами з обох доменів — гри і бота. Якщо подія `RaidReturn` зі світу Chibigochi не лізе в нашу модель Event, це означає, що модель неправильна, і ми виявляємо це на стадії документа, до коду.

Чому бот першим, а не гра: цикл feedback'у у бота тісніший. Розробник пише повідомлення, бачить, чи ядро прийняло подію, наступного дня перевіряє recall. У грі цикл "запустив сцену, зіграв момент, перезавантажив, перевірив" довший. На стадії формування контрактів швидший feedback дає вищу якість контрактів. PyO3 — обкатана зріла технологія. GDExtension через `godot-rust` працює, але має мобільні нюанси (Android NDK, arm64, статичне лінкування), і ловити їх паралельно з відлагодженням ядра системи — означає ловити дві категорії багів одночасно. Послідовно — ловимо по одній.

## 4. Подія (Event)

Подія — єдиний канал входу інформації у ядро системи. Хост-адаптер створює подію зі свого доменного об'єкта і викликає `engine.ingest(event)`. Подія потрапляє у шар Сесія і може отримати попередню оцінку ваги.

### 4.1 Обов'язкові поля події

- **type** — рядковий тип події у домені хоста. Наприклад: `user_message`, `assistant_reply`, `raid_return`, `gift_received`, `topic_shift`, `manual_note`. Ядро системи не валідовує конкретний список — він належить хосту. Але тип потрібен для scoring, фільтрації і recall.
- **source** — хто або яка система створила подію. Наприклад: `telegram_user_42`, `godot_dialogue_node`, `bot_command_layer`, `system_lifecycle`.
- **timestamp** — момент події в UTC, точність мілісекунди.
- **session_id** — ідентифікатор поточної сесії, до якої належить подія. Формат — рядок, який хост вибирає сам (типовий шаблон: `YYYY-MM-DD_NNN`).
- **payload** — корисне навантаження події. Структура належить хосту: для `user_message` це текст і метадані; для `raid_return` — результат рейду, отримані предмети, втрати. Ядро системи не інтерпретує payload, але зберігає його для recall і компресії.
- **schema_version** — версія схеми події, яку використовує адаптер. Дозволяє ядру системи правильно обробляти події після оновлень адаптера.

### 4.2 Опціональні поля події

- **tags** — список тегів, що хост уже знає для цієї події. Наприклад: `["care", "healing"]` для епізоду в грі, або `["work", "deadline"]` для бота. Теги — головний механізм фільтрації на стадії 1 recall, тому їх важливо передавати, де хост уже їх має.
- **theme** — рядкова тема, до якої належить подія, якщо хост її знає. Наприклад: тема обговорення в боті, тема дня в грі.
- **emotional_tone** — попередня оцінка емоційного тону події хостом, якщо доступна. Наприклад: `concern`, `joy`, `neutral`. Не обов'язково, але корисно для scoring.
- **links** — список зв'язків із попередніми подіями або архівними спогадами. Наприклад: `[{"kind": "follow_up", "target": "event:abc123"}]`.
- **importance_hint** — підказка про важливість, яку хост хоче передати ядру. Не остаточна оцінка — ядро системи має свої правила scoring, але hint враховується.
- **processing_mode** — `immediate` чи `defer_to_sleep`. Дозволяє хосту вказати, що подія має пройти повну обробку зараз або може почекати до sleep.

### 4.3 Приклади подій з обох хостів

**Telegram-бот, користувач написав факт про себе:**
```

timestamp: 2026-05-17T19:32:11.420Z
session_id: 2026-05-17_005

```

**Chibigochi, героїня повернулась із рейду:**

**Chibigochi, гравець лікує героїню після рейду:**

## 5. Шари пам'яті

### 5.1 Шар Сесія

Зберігає робочу пам'ять поточної взаємодії. У грі — живий день; у боті — поточна розмова; у третьому проєкті — поточна робоча сесія.

**Зберігається у файлах**: `memory/sessions/<session_id>/events.jsonl` (активний хвіст нормалізованих подій у порядку надходження), `memory/sessions/<session_id>/archived/events-<NNN>.jsonl` (raw-події, вже покриті Complete-архівами) і `memory/sessions/<session_id>/session.md` (читабельний для людини текстовий виклад). Markdown-вигляд лишається повним людським журналом; кодове читання активного контексту йде через `events.jsonl`, а evidence/audit-шляхи можуть читати archived-сегменти.

**Має ліміт розміру.** Конкретні цифри (кількість подій, символів, токенів) задаються конфігом хоста, не зашиті в ядро. При наближенні до ліміту ядро системи попередньо стискає найстаріші частини у короткий summary, який залишається в сесії, а оригінальні події переходять у буфер під sleep. Живий хвіст сесії залишається повним — це він підтримує відчуття безперервності в LLM-контексті.

**Підтримує ранню фіксацію важливих подій.** Якщо подія прийшла з `importance_hint: high` або scoring дав їй високу попередню вагу, ядро системи може ще до sleep записати її як preliminary archive entry. Це страхує від втрати важливих моментів при раптовому обриві.

**Консолідація сесії** виконується через sleep-run driver (`begin_sleep_run` → `next_sleep_batch` → `submit_sleep_batch` → `finish_sleep_run`). Complete sleep покриває старші події archive entry через `source_event_ids`, лишає свіжий active tail у `events.jsonl`, а raw-події, які вже покриті Complete-архівами і старші за active tail, може ротувати у `sessions/<session_id>/archived/events-<NNN>.jsonl`. Покриття Complete-архівів кешується в `session.json` як `archived_to`, `archived_event_ids` і `archived_event_index_complete`, щоб `core_context_package` і sleep selection не сканували весь архів на кожен хід; для старих metadata існує rebuild path із Complete archive entries. Для prompt context такі події більше не вважаються короткостроковою памʼяттю: `core_context_package` подає їх через `long_memory`, а `short_memory` містить тільки unarchived active tail. Evidence/audit paths читають active + archived raw segments з дедуплікацією за `event_id`; `session.md` лишається повним людським журналом.

### 5.2 Шар Архів

Зберігає довгострокові спогади, які пройшли відбір під час sleep. Це не сухий протокол подій — це стиснений досвід. Для Chibigochi це означає, що архівний спогад має нести емоційний сенс пережитого дня, а не лист механік. Для бота — підсумок теми, рішень і важливих фактів про користувача. Для робочого проєкту — рішення, причини змін, виправлення.

**Зберігається у файлах**: `memory/archive/<YYYY>/<MM>/<spogad_id>.json` (pretty-printed JSON: машинно-читабельне ядро запису і поточна human-inspectable форма). Окремий Markdown-виклад архівного спогаду відкладено до реального workflow читання архіву людиною; зараз його не пише `FileStorage`. Розбиття на роки і місяці дає природну часову навігацію.

**Архівний спогад містить як мінімум:**

- ідентифікатор і версію схеми;
- час події (або діапазон, якщо спогад покриває кілька подій);
- джерело сесії, з якої народився;
- тему (theme) і теги;
- короткий зміст (`gist`) — одне-два речення;
- емоційний наратив (`narrative`) — стиснений текст пережитого досвіду;
- ключові факти (`facts`) — точні твердження, що мають довготривалу цінність;
- цитати (`quotes`) — короткі дослівні фрагменти, що зберігають голос моменту;
- вагу (`weight`) — числове значення значущості;
- актуальність (`freshness`) — числове значення, що зменшується з часом через decay;
- зв'язки (`links`) — посилання на інші архівні спогади або на елементи шару Ядро;
- `embedding_model_id` — порожнє на v0.1, заповнюється коли вмикається стадія 2 recall;
- `embedding` — порожнє на v0.1, заповнюється тоді ж.

Тут "як мінімум" — рівень обов'язкових полів. Точна JSON-схема буде в окремому документі, але саме ці поля несуть стратегічний сенс: гра має могти прочитати з архіву "пережитий день", бот має могти прочитати "стійкий факт про користувача", і recall має могти ранжувати релевантність по тегах, темі, вазі, часі і зв'язках.

**Ваги і decay.** Кожний архівний спогад має вагу і актуальність. Вага задається під час sleep і може зростати при повторних зверненнях (recall hit), зв'язках із новими подіями, участі у формуванні candidate belief. Актуальність зменшується з часом за конфігурованою функцією decay (на v0.1 — проста експоненційна або лінійна, конкретний вигляд задасть подальший документ). Спогад із низькою актуальністю не зникає — він просто отримує нижчий ранг у recall і може бути сильніше стиснений під час майбутніх consolidation passes.

**Захист від втрати ультраважливих спогадів.** Якщо спогад має `weight >= weight_floor_critical` (значення задається конфігом), його decay сповільнюється або зупиняється. Це не autopromotion у шар Ядро — це "цей спогад не зникне з recall сам по собі".

### 5.3 Шар Ядро

Стабільна основа пам'яті. Для Chibigochi — те, що героїня пам'ятає як частину себе: ім'я, ставлення до гравця, фундаментальні факти, сталі межі особистості. Для бота — довготривалі факти про користувача, його сталі переваги, важливі цілі. Для робочого проєкту — стійкі принципи, рішення, які впливають на майбутні рішення.

**Adaptive Stable Core.** Шар Ядро має бути живим, але не нестабільним. Він адаптується через validated memory units, reflection candidates, recall feedback і contested/deprecated lifecycle, але не приймає прямі висновки LLM як істину. Будь-яка зміна Core має пройти структурний шлях: джерело -> стиснення -> fidelity validation -> кандидат або explicit review -> підтверджене оновлення.

Це означає, що Core може змінюватися під час реальної експлуатації: нові стабільні факти з'являються, старі можуть стати contested, рутина виходить із активного recall. Але стабільність сильніша за швидкість: випадкова фраза, невдала компресія або один фантазійний висновок агента не повинні переписати довготривалу основу.

**Зберігається у файлах**: `memory/core/store/<category>.json` (наприклад, `name.json`, `pet.json`, `preference.json`). Це pretty-printed JSON, який є і машинним форматом, і поточною human-inspectable формою. Окремі `<category>.md` dumps не пишуться у v0.2; якщо зʼявиться реальний workflow ручного читання/редагування Core поза debug-командами, їх можна додати як похідний export, а не як нову правду. Розбиття на категорії дозволяє людині зайти і прочитати "що ядро знає про користувача в категорії X".

**Розрізнення Core Store і Core Context Package.** Core Store — це сама стабільна основа, яка живе у файлах і змінюється повільно через промоцію. Core Context Package — це готовий пакет для LLM, який ядро системи збирає на запит хоста, поєднуючи Core Store, живу сесію, релевантний Архів і поточний доменний стан, що його хост передає в момент запиту. Поточний стан (фізичний стан героїні зараз, активна тема розмови зараз, поточне завдання у проєкті) **не зберігається в шарі Ядро**. Він приходить від хоста у момент збору пакету.

На v0.1 `engine.core_context_package(request)` повертає:

- `core_facts` — стабільні факти Core Store з поточного `core_scope`; категорії є вільними normalized `snake_case`, а не закритим списком;
- `session_recent` — короткий хвіст активної сесії;
- `session_trace` — ширший trace подій поточної сесії;
- `archive_relevant` — результат stage1 recall по Архіву;
- `domain_state` — стан, який хост передав у запиті.

Хости не повинні самі дублювати збір recent/trace/archive-контексту. Вони формують доменний стан, викликають `core_context_package`, а потім передають пакет у свій LLM prompt.

**Prompt geometry.** Хост має збирати LLM prompt так, щоб модель чітко бачила межі шарів, а не отримувала один змішаний текст. Мінімальна форма:

1. `system_prompt` — правила поведінки і роль.
2. `domain_state` — поточний стан хоста: хто говорить, активна сесія, час, системні прапори, режим памʼяті.
3. `core_memory` — шар Ядро: стабільні confirmed факти і потрібні contested/deprecated позначки.
4. `long_memory` — довга памʼять: релевантні archive memory units / compact projection, без raw JSON-аудиту.
5. `short_memory` — коротка памʼять: unarchived active tail поточного діалогу.
6. `current_user_message` — поточна репліка користувача, явно відокремлена від історії.
7. `assistant_response_slot` — місце, де має початися відповідь асистента.

Це потрібно, щоб бот не втрачав геометрію розмови: де стабільне знання, де давній спогад, де живий діалог, де завершився transcript і де має бути нова відповідь. Prompt assembly не має зливати `core_memory`, `long_memory` і `short_memory` в один неструктурований список.

LLM-facing render цього пакета має бути структурно видимим: XML-подібні теги або такі самі чіткі секції. Наприклад, `<core_memory>...</core_memory>`, `<long_memory>...</long_memory>`, `<short_memory>...</short_memory>`, `<current_user_message>...</current_user_message>`. `telegram_chat_system.md` має прямо пояснювати моделі: `long_memory` — це знання з минулих розмов, `short_memory` — поточний живий діалог, `current_user_message` — щойно отримана репліка, відповідати треба саме на неї, враховуючи попередні блоки. Це структурний сигнал, не доменний хардкод.

**Бюджет контексту для живого використання.** Цільовий benchmark для host prompt — максимум **11k токенів** на памʼять у запиті. Розподіл:

- до **7k токенів** — поточна памʼять: `session_recent`, `session_trace` і активний domain/current turn context;
- до **3k токенів** — стиснута памʼять: `archive_relevant`, тобто вже сконсолідовані archive memories;
- до **1k токенів** — шар Ядро: `core_facts`.

Це не дозвіл просто обрізати текст до ліміту. Стиснення має переносити сенси: емоційні маркери, персональні сигнали, причинно-наслідкові переходи тем, важливі рішення, виправлення і невирішені питання. Якщо пакет не вміщується в бюджет, система має зменшувати обсяг за правилами значущості й релевантності, а не випадково викидати тихі особисті моменти. На v0.1 `core_context_package` має детермінований budget allocator із консервативним estimator-ом і `budget` report у відповіді. Точний provider-specific tokenizer лишається задачею host-рівня або v0.2.

**Компактне prompt-facing представлення.** Канонічні файли памʼяті мають бути повними, аудитними і зручними для відлагодження: там допустимі довгі `event_id`, `archive_id`, `core_fact_id`, `task_id`, schema versions, source metadata і звʼязки. Але LLM prompt — це інша межа. У prompt не можна без потреби тягнути довгі технічні ідентифікатори, дубльовані schema-поля, службові source names або числові хвости, які не допомагають моделі відповісти. Такі поля витрачають token budget і працюють проти головної мети стиснення. Принцип: **storage/debug view може бути повним; prompt view має бути семантично достатнім і компактним**. Технічні ID потрапляють у prompt тільки якщо вони потрібні для команди користувача, audit/debug режиму або конкретної операції на кшталт `/core_update <id>`.

На v0.1 compact projection перед chat prompt є спільним контрактом ядра: `render_memory_view(package, current_user_message)` прибирає службові ID, schema/source/debug metadata і повертає LLM-facing блоки `core_memory`, `long_memory`, `short_memory`, `current_user_message` та `assistant_response_slot`. Повний `core_context_package` лишається API/debug формою. Хости не повинні дублювати projection policy; вони передають пакет у ядро, отримують memory view і склеюють його зі своїм system prompt та provider call.

**Token telemetry.** Host має логувати не тільки факт, що prompt вклався в 11k, а й реальну вартість звернень до моделей. Для Gemini Telegram host пише provider `usageMetadata` по кожному LLM-виклику, estimated prompt/output tokens, baseline "raw history without compression" для chat turn, savings estimate і raw→stored archive / raw→prompt archive / raw→compact memory ratios для sleep. Це потрібно, щоб довести економію від стиснення, а не просто заявляти її.

Уточнення 2026-05-22: для Telegram host compact projection більше не є JSON-дампом навіть у скороченому вигляді. Звичайний chat prompt має рольовий transcript `user:` / `assistant:`, явно відокремлює поточне повідомлення користувача від попередньої історії та не дублює `session_recent` як частину `session_trace`. Архів у prompt подається через `compact_memory`: короткі LLM-написані тези типу "подія -> висновок", які бот має знати як природну памʼять. Multi-track поля (`emotional_markers`, `personal_signals`, `topic_thread`, `relational_tone`) лишаються у storage/debug archive для audit, Core bridge і майбутнього reflection, але не йдуть у звичайний chat prompt як JSON-проекція. Це зроблено, щоб модель не втрачала геометрію діалогу і щоб стиснення реально економило токени.

`core_context_package` читає всі активні файли `memory/core/store/*.json`, а не старий whitelist категорій. Це обовʼязково, бо Core Store у v0.1 має вільні normalized `snake_case` категорії від LLM-pass (`name`, `pet`, `physical_trait`, `food_preference`, тощо). Старий список `profile/preferences/relationship` лишається тільки як legacy seed у конфігу, але не обмежує читання Core facts.

### 5.4 Канали Запису В Core Store

Core Store має кілька різних каналів впливу. Їх треба не змішувати, бо вони мають різний рівень довіри і різну мету.

**Explicit user path** — активний у v0.1. Користувач явно дає команду зберегти факт, наприклад `/remember Користувача звати Микита.`. Хост передає цей факт у `engine.upsert_core_fact(input)`. Це не regex extraction: рішення записати факт приймає користувач або власник системи.

Explicit path обовʼязково має задавати `scope`, якщо host працює з кількома користувачами, чатами або персонажами. У Telegram host scope дорівнює session id `telegram_<chat_id>`. `core_context_package` повертає тільки факти поточного `core_scope`, тому факти з одного чату не мають потрапляти в інший.

**Heuristic tagging path** — активний у v0.1, але не пише в Core Store. Хост може додати event-теги (`personal_fact_signal`, `name_reference`, `age_reference`, `preference_signal`, `explicit_memory_request`) і підняти `importance_hint`, щоб sleep/reflection пізніше уважніше переглянули подію. Це мʼякий сигнал, не рішення "це Core-факт".

**Archive personal-signal bridge** — активний у v0.1. Після успішного sleep LLM personal-signal pass виділяє стабільні self-statements і user-specific facts. Host не парсить raw text і не має whitelist-а тем; він тільки застосовує прозорі gating rules: `confidence >= 0.85`, supporting `source_event_ids` містять user-authored event, free category нормалізується до `snake_case`, і немає близького дубля в активному Core. Це bridge із Archive в Core, а не regex extraction.

**Reflection-based promotion path** — активний у v0.2. Ядро переглядає validated memory units і активний Core, знаходить повторювані теми й потенційні висновки, створює `CandidateBelief`, перевіряє source provenance і суперечності, після чого власник підтверджує або відхиляє промоцію в Core Store. Цей шлях потрібен для висновків типу "користувач постійно повертається до теми X", а не для прямих профільних фактів.

Regex extraction із plain text напряму в Core Store заборонений як основний шлях. Він створює демо-ефект, але не моделює консолідацію памʼяті і засмічує шар Ядро. Core поповнюється явно через `/remember`, через Archive → Core bridge на LLM-витягнутих `personal_signals`, які пройшли gating, або через reflection candidate lifecycle з manual review.

Core-факт має керований lifecycle. Host може викликати `engine.patch_core_fact(input)`, щоб оновити текст або змінити статус. Telegram-команда `/core_forget <id>` ставить `deprecated`, а `/core_update <id> <text>` оновлює текст у поточному `core_scope`. Approved reflection candidate може позначити суперечливий active факт як `contested`. `Contested` факти лишаються видимими в context із marker; `Deprecated` факти залишаються у JSON як історія, але не повертаються в `core_context_package`.

**Кандидати на промоцію (candidate beliefs).** Reflection-процес періодично переглядає Архів і виділяє повторювані теми, патерни, потенційні стабільні висновки. Вони зберігаються у `memory/core/candidates/<candidate_id>.json` як проміжний стан. Кандидат не потрапляє в Core Store автоматично.

**Правила candidate promotion** мають бути структурними, а не тематичними або кількісно-хардкодними. LLM-агент може вільно виділяти зміст, але кожен кандидат має мати:

1. атомарну тезу, яка не змішує кілька незалежних висновків;
2. явні `source_memory_unit_ids` або `source_archive_ids`;
3. коротке пояснення, чому це не тимчасовий стан, а потенційно стабільний висновок;
4. оцінку ваги/важливості, сформовану агентом, без фіксованої кількості обов'язкових джерел;
5. результат fidelity/consistency review: чи не перекручено джерела, чи немає суперечності з активним Core.

Агенти не "правлять істину" напряму. Вони створюють, критикують, стискають і маршрутизують кандидатів. Core Store змінюється тільки через керований lifecycle: `candidate` → `ready_for_review` → `confirmed`/`rejected` → `active CoreFact`. На першій v0.2-ітерації підтвердження ручне через host-команду; auto-confirm переноситься на пізніший етап після реальної статистики якості.

## 6. Recall

### 6.1 Recall Query

Запит на пошук релевантних спогадів. Хост передає контекст і параметри, ядро повертає список елементів.

**Поля запиту:**

- **context** — поточний контекст запиту: активна сесія, активна тема (якщо є), останній фрагмент розмови чи дії. Текст або структура — формат залежить від типу запиту.
- **query_text** — необов'язково, текстовий запит, якщо хост хоче explicit пошук ("згадай, що користувач казав про роботу").
- **filters** — словник обмежень: часове вікно, теги, тема, мінімальна вага, мінімальна актуальність, типи спогадів (наприклад, тільки `personal_fact`).
- **limit** — максимум елементів у відповіді (типове v0.1 значення — 5-10).
- **include_core** — булеве, чи додавати релевантні факти з Core Store до відповіді.
- **explain** — булеве, чи додавати поле `relevance_explanation` до кожного елемента (важливо для debug і для довіри).

### 6.2 Recall Result

Список елементів, де кожен містить:

- ідентифікатор архівного спогаду або Core-факту;
- короткий зміст (`gist`);
- наратив (для архівних спогадів);
- факти (для Core-елементів);
- цитати, якщо є;
- джерело і час;
- теги і тему;
- вагу і актуальність;
- `relevance_score` — числове ранжування, що показує, чому елемент потрапив у видачу;
- `relevance_explanation` — рядкове пояснення (наприклад, "тег `personal_fact` збігається з запитом, вага 0.82, актуальність 0.91").

Recall завжди детермінований на стадії 1. Той самий запит на тих самих даних повертає той самий результат. Це важливо для тестів і для довіри.

Якщо запит має `session_id`, archive candidates обмежуються записами з відповідним `source_session_id`. Це частина isolation boundary, а не scoring preference. Без цього свіжа сесія може отримати чужу або стару історію як релевантний archive context. Для явного глобального пошуку хост має не передавати `session_id`.

### 6.3 Трирівневий пайплайн

**Стадія 1 (активна v0.1).** Структурний фільтр + scoring.

Послідовність кроків:

1. Звуження за фільтрами (час, теги, тема, типи) — отримуємо набір кандидатів.
2. Обчислення `relevance_score` для кожного кандидата за формулою. Базова формула v0.1:
   `score = weight × freshness × theme_match_factor × tag_overlap_factor × link_bonus`.
   Конкретні коефіцієнти задаються конфігом і документуються окремо.
3. Сортування за `relevance_score` за спаданням.
4. Обрізання за `limit`.
5. Додавання `relevance_explanation` за прапором `explain`.

**Стадія 2 (v0.3+ opt-in).** Embedding re-ranking над топ-K зі стадії 1.

Якщо вмикається в конфігу і архівні спогади мають embeddings: query embed'иться, обчислюється косинусна близькість до кожного кандидата, відбувається перебудова рангу через зважену комбінацію `score_stage1` і `cosine_similarity`. K — окремий параметр (типове значення — 30-50).

Embeddings обчислюватимуться для validated memory units або archive memories тільки якщо host/owner увімкнув відповідну feature. Поле `embedding_model_id` дозволяє виявити, що частина архіву embed'ена старшою моделлю — і запустити re-embed-міграцію через LLM/vector tasks.

**Стадія 3 (v0.3+ opt-in).** LLM re-ranking.

Якщо вмикається: топ-N зі стадії 2 (наприклад, N=10) передається LLM через `PendingTask` типу `recall_rerank` із промптом, який просить LLM відсортувати кандидатів за справжньою релевантністю до запиту і повернути пояснення. Результат — фінальний ранг.

Архітектурно ці три стадії — окремі компоненти `RecallStage1`, `RecallStage2`, `RecallStage3` усередині ядра системи. Вмикання стадій 2 і 3 змінює конфіг, не код.

## 7. PendingTask: як ядро системи просить LLM

### 7.1 Форма задачі

`PendingTask` — це структура, яку ядро системи повертає викликаючому коду замість блокування на мережевому виклику. Усі дані, потрібні для виконання задачі, лежать всередині.

**Поля задачі:**

- **task_id** — унікальний ідентифікатор, серіалізовний.
- **task_type** — тип задачі: `sleep_compression`, `compact_memory_pass`, `memory_unit_pass`, `memory_fidelity_pass`, `reflection_analyze`, `forget_review_pass`, `recall_rerank`, `compute_embedding`, `fact_check`, `tag_proposal`, тощо. На v0.2 реально використовуються sleep/memory-unit/fidelity/reflection/forgetting задачі; embeddings і LLM rerank лишаються opt-in майбутніми типами.
- **prompt_id** — ідентифікатор промпта з папки `prompts/`. Ядро системи не вбудовує текст промпта в код; промпт читається з файлу за `prompt_id` (точна механіка — у розділі 12).
- **role_hint** — рекомендована роль моделі: `reasoning`, `balanced`, `fast`. Хост вибирає конкретного провайдера і конкретну модель для цієї ролі за своїм конфігом.
- **inputs** — структурований вхід для задачі: текст сесії, що стискається; список архівних спогадів для re-rank; подія для scoring.
- **expected_output_schema** — опис очікуваної форми результату: JSON schema name для structured passes або plain/prose contract для текстових overlay-проходів.
- **created_at** — час створення задачі.
- **state** — `pending`, `submitted`, `completed`, `failed`. Ядро змінює state при відповідних операціях.

### 7.2 Resume і tracking

Хост викликає `engine.resume(task_id, result)` або `engine.resume_failed(task_id, error_info)`. Ядро системи:

1. Валідовує `result` проти `expected_output_schema`.
2. Якщо валідація пройшла — застосовує результат до пам'яті (для `sleep_compression` — створює архівний спогад; для `score_event` — оновлює вагу; для `recall_rerank` — повертає викликаючому recall новий ранг).
3. Помічає task як `completed`.

Якщо валідація провалилась — задача залишається у `submitted`-стані, ядро повертає помилку викликаючому. Хост може повторити з виправленим результатом.

### 7.3 Серіалізація і lifecycle resilience

Усі pending tasks зберігаються у `memory/tasks/<task_id>.json` одразу при створенні. Стан `state` оновлюється у файлі при кожній зміні. При запуску ядро системи читає `memory/tasks/`, бачить незавершені задачі і повертає їх список хосту через `engine.pending_tasks()`. Хост вирішує, які з них продовжувати виконувати.

Це дає очевидну поведінку при обриві: застосунок вбили посеред sleep — ми не втрачаємо ані сесію (вона у файлах), ані задачу (вона у `memory/tasks/`). При наступному запуску ядро каже хосту "є незавершений `sleep_compression` для сесії 2026-05-17_001", і хост вирішує: виконати зараз, відкласти, відкинути.

### 7.4 Типи задач на v0.2

На v0.2 ядро системи реально використовує такі LLM-backed задачі:

- **sleep_compression / consolidator prose overlay** — LLM-доробка архівного спогаду для storage/debug audit; структура збирається ядром із валідованих треків, а LLM дає prose gist/narrative overlay.
- **compact_memory_pass** — prompt-facing компресія сесії у короткі plain-text тези "подія -> висновок". Вихід — текстове поле `compact_memory`, яке потім повертається в ordinary chat prompt замість JSON-проекції full archive.
- **memory_unit_pass** — розбиває sleep/archive material на atomic memory units із тезами, source ids, weight і status.
- **memory_fidelity_pass** — перевіряє memory unit проти evidence pack і повертає fidelity verdict.
- **reflection_analyze** — працює по validated memory units і Core, створює candidate beliefs.
- **forget_review_pass** — переглядає old/low-signal units і рекомендує keep/protect/forget; engine hard-gate вирішує, що реально можна позначити Forgotten.
- **score_event** (опціонально) — попередня оцінка значущості події. Вхід — подія і короткий контекст. Вихід — числова вага і коротке обґрунтування. Може бути вимкнена, тоді scoring повністю детермінований.

Решта типів (`recall_rerank`, `compute_embedding`, `fact_check`, `tag_proposal`) задекларовані як майбутні opt-in tasks. Коли вони знадобляться, форма вже задана.

## 8. Sleep як двостадійний процес

Sleep — процес консолідації сесії у архівний спогад. На v0.1 виконується у дві стадії: алгоритмічна частина без LLM, потім LLM-доробка через `PendingTask`. Це дає важливу властивість: пам'ять жива і без живого LLM-адаптера, просто із сухішими архівними спогадами.

### 8.1 Стадія 1: алгоритмічна (синхронна, без LLM)

Виконується одразу при виклику `engine.sleep(session_id)`. Кроки:

1. Зчитати активні події сесії з `memory/sessions/<id>/events.jsonl`.
2. Виконати дедуплікацію і базову нормалізацію.
3. Виконати scoring (детермінований за подіями, або з використанням `score_event` PendingTask'у, якщо ввімкнено).
4. Відібрати події для стискання тільки серед тих подій сесії, які ще не входять у жоден archive entry цієї сесії.
5. Якщо unarchived window достатньо великий (`partial_sleep_min_events`, default `10`), залишити найсвіжіший active tail незаархівованим. Default `active_tail_ratio` — `0.30`: при sleep старша частина переходить в archive, а найсвіжіша частина лишається в `session_recent` / `session_trace`.
6. Серед compactable-подій застосувати `sleep.max_events` (default `80`). Якщо compactable-подій більше за `sleep.max_events`, пріоритет мають вища вага і новіші події при рівній вазі. Це не замінює майбутній агентивний salience scoring, але не дозволяє раннім рівноважним повідомленням витіснити пізніші особисто значущі частини діалогу.
7. Згрупувати їх за темами або за часовою близькістю.
8. Сформувати **попередній архівний спогад** із базовим summary, повним списком тегів, фактами і вагою. Цей попередній спогад зберігається у Архів одразу — він уже корисний для recall, навіть якщо LLM-доробка ніколи не відбудеться.
9. Повернути `PendingTask` типу `sleep_compression` для LLM-доробки full archive і окремий `PendingTask` типу `compact_memory_pass` для коротких prompt-facing тез.

### 8.2 Стадія 2: LLM-доробка через PendingTask

Хост виконує `sleep_compression` через свого LLM-провайдера, використовуючи промпти із `prompts/`. Мінімальний baseline — один `sleep_compression.md`; Telegram host у v0.1 за замовчуванням виконує multi-pass sleep із окремими prompt-файлами для emotional/topic/personal/relational/consolidator проходів. Результат — повноцінний архівний спогад: емоційний наратив, відібрані цитати, уточнені факти, скоригована вага і multi-track поля.

Паралельно хост виконує `compact_memory_pass` через `prompts/compact_memory_pass.md`. Це окрема LLM-задача над тим самим raw sleep input, а не програмна проекція з JSON і не `take(N)` / `truncate(N chars)`. Її результат — змінна кількість коротких plain-text тез "подія -> висновок", які зберігаються в `ArchiveEntry.compact_memory`. Кількість тез визначає агент за кількістю змістовних memory units у розмові: одна тема може дати одну тезу, кілька різних епізодів — кілька тез, рутина може не дати жодної окремої тези. Саме ці тези є prompt-facing стиснутою памʼяттю для звичайної відповіді бота.

Хост повертає результати через `engine.resume_compact_memory_pass(task_id, text)` і `engine.resume_sleep_compression(task_id, result)`. Ядро системи **оновлює** архівний спогад, що його створено у стадії 1: записує `compact_memory`, замінює базовий summary на повний наратив, додає цитати, оновлює вагу і теги. Сам ідентифікатор спогаду зберігається, тому будь-які зв'язки, що вже на нього посилались, не ламаються.

### 8.3 Покриття подій замість очищення сесії

Після успішного завершення sleep Complete-archive-covered raw-події ротуються з активного `events.jsonl` у `sessions/<id>/archived/events-<NNN>.jsonl`. `session.md` лишається повним людино-доступним журналом. Archive entry містить `source_session_id` і `source_event_ids`, які показують, які саме події вже сконсолідовано. `session.json` підтримує перебудовний coverage-cache: `archived_to` містить complete archive ids, `archived_event_ids` — покриті raw event ids, `archived_event_index_complete` показує, чи можна довіряти cache без archive scan.

Це важлива межа: raw storage може містити стару розмову, але prompt context не має дублювати її одночасно як raw session events і як archive memory. `core_context_package` читає активний хвіст із `events.jsonl` і додатково фільтрує archived events із `session_recent` та `session_trace`; якщо хосту потрібен поточний turn, він передає його окремо у `domain_state` або в власному chat prompt. Evidence/fidelity і Core bridge можуть читати archived-сегменти за `source_event_ids`.

### 8.4 Sleep Triggers

Manual sleep лишається публічним викликом `engine.sleep(session_id)`. Автоматичний sleep у v0.1 не привʼязаний до кількості повідомлень. Продуктові тригери тільки такі:

1. **Budget-pressure sleep.** Коли `core_context_package` показує, що active session/current memory наближається до token budget, хост ставить `engine.sleep(session_id)` у background queue. Мета — не дати prompt роздутися і перевести старшу частину живої розмови в compact archive memory.
2. **Scheduled idle sleep.** Хост запускає sleep за нічним розкладом у період типової неактивності, наприклад о 04:00 локального часу, якщо сесія має незаархівовані події і певний час не отримувала нових повідомлень.

Дедуплікація належить хосту і ядру разом: якщо для сесії вже є pending/submitted `sleep_compression`, новий sleep не ставиться. Якщо preliminary archive уже покрив частину подій, ці події більше не рахуються як незаархівовані і не мають повторно потрапляти в наступний `engine.sleep(session_id)`.

Хост не стискає памʼять сам. Хост тільки:

1. дивиться на token budget report або нічний idle schedule;
2. викликає `engine.sleep(session_id)`, коли один із продуктових тригерів спрацював;
3. виконує `compact_memory_pass` і `sleep_compression` через свій LLM-провайдер;
4. повертає результати через `resume_compact_memory_pass` і `resume_sleep_compression`.

### 8.5 Multi-pass compression

Початковий `sleep_compression` був baseline: один `PendingTask`, один LLM-прохід, один `ArchiveEntry`. Це достатньо, щоб довести foundation: ingest -> sleep -> archive -> recall -> context package. Але це не фінальна когнітивна модель памʼяті.

Проблема одного проходу: модель природно стискає розмову як тематичний звіт. Вона добре памʼятає "великі" теми на кшталт літаків або колонізації, але може втратити тихий особистий епізод, який для живої памʼяті важливіший за енциклопедичну тему. Це не виправляється хардкодом доменів. Потрібна інша структура sleep.

Multi-pass sleep у v0.2 оркеструється ядром через pull-based driver: ядро створює preliminary archive, durable `SleepRun`, extraction/consolidation batches і typed `LlmRequest`-и; host тільки виконує prompt → text і повертає responses. Спеціалізовані LLM-проходи створюють emotional/topic/personal/relational/memory-unit material, а ядро детерміновано збирає archive result і накладає consolidator prose overlay.

Це завершує колишній host-level multi-pass борг: graph/order/retry/fail-soft/fallback живуть у Rust core, prompt loading і provider calls лишаються в host.

**Emotional pass.** Шукає емоційно заряджені моменти: привʼязаність, тепло, гумор, фрустрацію, гордість, занепокоєння, ностальгію, цікавість. Повертає `emotional_markers` з `target`, `affect`, `strength`, `source_event_ids` і коротким evidence. Важливо: код не знає, які сутності важливі. Значущість визначається через події і LLM-pass.

**Topic thread pass.** Зберігає епізодичну структуру: які теми йшли в якій послідовності, як одна тема переходила в іншу, де енергія діалогу зростала або спадала. Це не плоский список тем, а thread.

**Personal signal pass.** Витягує, що нового стало відомо про користувача як людину: люди, тварини, місця, цінності, інтереси, звички, самоопис, важливі обʼєкти, повторювані патерни. Це джерело для майбутніх Core candidates.

**Relational pass.** Фіксує, як розмова вплинула на стосунок між користувачем і агентом: warmth, distance, trust, playfulness, tension, shared joke, correction, style of address.

**Consolidator.** Бере результати попередніх проходів і збирає один або кілька archive entries. Порядок пріоритету інший, ніж у wiki-summary: emotional markers і personal signals задають центр спогаду; topic facts стають контекстом, а не навпаки.

**Compact memory pass.** Окремо від consolidator читає raw події sleep window і повертає короткі тези "подія -> висновок", які людина природно пригадала б через тиждень. Це не audit-запис і не enrichment: це власне prompt-facing компресія. Вона зберігається у `compact_memory` і саме вона йде в `archive_relevant` для ordinary chat.

Це означає multi-track archive format. Старі поля `gist`, `narrative`, `facts`, `quotes`, `tags`, `theme`, `weight` лишаються для сумісності. Додаються окремі треки:

```text
compact_memory
emotional_markers[]
topic_thread[]
personal_signals[]
relational_tone
```

Recall v0.2 має вміти діставати як весь archive entry, так і конкретний трек: наприклад, "personal_signals за останній тиждень" або "emotional_markers з високою strength".

На v0.1 звичайний `archive_relevant` для chat context пріоритезує `compact_memory`: якщо поле є, recall item повертає його як `gist`, додає `compact_memory` і не тягне `narrative`, `facts`, `quotes` у prompt-facing форму. Повні multi-track дані лишаються у storage і доступні через debug-команди на кшталт `/archive_last`.

**Rolling / partial sleep.** v0.1 уже не поводиться як "взяти всю сесію й очистити". Коли unarchived window достатньо великий, ядро стискає старшу частину active session, а молодший tail лишає активним. Конкретна форма: `active_tail_ratio` і `partial_sleep_min_events` у `SleepStage1Config`. Default — 30% хвоста лишається active, але для коротких ручних `/sleep` менших за `partial_sleep_min_events` стискається все. Це дає плавну памʼять: свіже лишається детальним у session, старіше переходить в archive.

## 9. Reflection і review lifecycle

Reflection — окремий від sleep процес довшого осмислення Архіву. Не запускається автоматично на кожне завершення сесії. Запускається явним викликом хоста (`begin_reflection_analysis` / submit response) або ручною командою власника.

У v0.2 reflection реалізований як pull-based LLM task:

- Вхід: validated `memory_units`, compact archive summaries і поточний Core context.
- Робить: повертає `LlmRequest` / task типу `reflection_analyze` з підбіркою спогадів.
- Хост виконує: LLM аналізує підбірку, повертає список кандидатів на стабільні висновки і список виявлених патернів.
- Результат: створення `candidate belief` записів у `memory/core/candidates/`.

Критичний інваріант: reflection не може напряму писати в Core. Він створює candidate beliefs; Core змінюється тільки через `review_candidate(approved)` або explicit Core API. Якщо кандидат суперечить active Core facts, approved review позначає старі факти як `contested`, не перезаписує їх мовчки.

Reflection працює не напряму по raw events, а по archive tracks, створених multi-pass compression:

- `memory_units`, якщо вони вже створені sleep/reflection pipeline;
- `compact_memory`;
- `emotional_markers`;
- `personal_signals`;
- `topic_thread`;
- `relational_tone`;
- старі сумісні поля `facts`, `quotes`, `tags`, `narrative`.

Основна одиниця — `memory_unit`, а не великий archive JSON. Reflection має бачити окремі короткі спогади з локальними мітками (`m1`, `m2`, `m3`) і повними source-id тільки у debug/storage. Це готує систему до майбутнього vector storage без того, щоб уже зараз його вмикати.

Перед тим як memory unit або candidate belief стане матеріалом для Core, він проходить fidelity validation. Validator перевіряє не важливість, а чесність трансформації: чи не додано того, чого не було у джерелі; чи не перебільшено емоційний тон; чи не випала ключова деталь; чи не переплутані ролі. Якщо unit викривлений, він має бути перероблений або лишитись debug/low-confidence матеріалом, але не ставати нормальним спогадом.

**Evidence pack для дорогих перевірок.** Fidelity validator і Core candidate reviewer не мають отримувати всю довгу розмову, якщо кандидат спирається на малий фрагмент. Memory unit зберігає `source_event_ids`; ядро системи формує мінімальний evidence pack: самі source events, повʼязані репліки, які потрібні для смислу, і короткий локальний контекст переходу теми. Host тільки виконує `LlmRequest`, який уже містить evidence inputs. Це не fixed window на N повідомлень. Evidence pack має бути настільки малим, наскільки дозволяє чесна перевірка, і достатнім, щоб велика модель могла сказати: "теза підтримана джерелом", "теза перебільшена" або "бракує важливої деталі".

Формування evidence pack є програмною збіркою з уже відомих source ids, не LLM-рішенням і не тематичним хардкодом. Базові правила:

1. Взяти всі `source_event_ids` memory unit або Core candidate.
2. Додати невелику кількість сусідніх подій навколо кожного source event для локального контексту; розмір вікна задається конфігом, а не зашивається в Rust.
3. Додати повʼязані quote/evidence фрагменти з `emotional_markers`, `personal_signals` або archive tracks, якщо вони прямо посилаються на ті самі source ids.
4. Вкластися у `evidence_pack_token_budget` (цільовий стартовий benchmark для Telegram host — до 1.5k токенів, але це конфіг, а не архітектурна константа).
5. Якщо місця бракує, пріоритет такий: source events -> direct evidence/quotes -> neighboring context.

На критичному шляху в шар Ядро дорога/розумна модель може бути виправданою, бо вона працює не з сотнями тисяч токенів raw history, а з маленьким evidence pack і одним або кількома кандидатами. Саме ця модель має формулювати фінальну коротку, щільну Core-тезу для review: без води, без переказу всього діалогу, але з достатнім сенсом і evidence.

**Model role routing.** Масові sleep/memory-unit задачі за замовчуванням ідуть у `ModelRole::Balanced`. Критичний шлях до Core має отримувати `ModelRole::Reasoning`: fidelity validation на evidence pack, core candidate review і фінальне формулювання Core-тези. Ядро системи задає тільки `role_hint`; конкретну модель, провайдера і ключ вибирає host config.

Promotion у Core має бути багаторівневою:

1. **Extraction/compression agent** створює memory units.
2. **Fidelity validator** перевіряє відповідність evidence pack, raw events або source archive.
3. **Layer/router agent** пропонує шар: archive, core_candidate, routine/discard або needs_revision.
4. **Core candidate reviewer** перевіряє, чи кандидат відповідає структурі Core: стабільність, конкретність, source evidence, відсутність явної суперечності, і формулює коротку щільну Core-тезу для review.
5. **Owner/host confirmation** переводить `ready_for_review` у активний CoreFact.

Це не доменний хардкод. Rust core не знає, що саме є "кішкою", "родиною" або "цінністю". Core бачить структурований кандидат, evidence, вагу, статуси review і рішення власника. Який саме спогад існує і як його назвати, вирішує LLM-pass; чи можна йому довіряти як трансформації — перевіряє validator; чи потрапить він у Core — вирішує lifecycle.

## 10. Storage у деталях

### 10.1 Розкладка файлів

```
memory/
  manifest.json              — версії, активні налаштування
  sessions/
    <session_id>/
      events.jsonl           — активний хвіст подій
      session.md             — людський виклад
      archived/
        events-<NNN>.jsonl   — raw-події, покриті Complete-архівами
  archive/
    <YYYY>/<MM>/
      <spogad_id>.json       — pretty JSON archive entry; current human-inspectable form
    units/
      <unit_id>.json         — атомарні memory units для reflection/recall/vector-ready майбутнього
    forgotten/               — reserved for future physical relocation; v0.2 forgetting uses reversible status flip
  core/
    store/
      <category>.json        — стабільні факти по категоріях; pretty JSON is current human-inspectable form
    candidates/
      <candidate_id>.json    — кандидати на промоцію
  tasks/
    <task_id>.json           — стан PendingTask'ів
  runs/
    sleep/
      <sleep_task_id>.json   — durable SleepRun orchestration cursor
  journal/
    <op_id>.json             — журнал мульти-файлових операцій
```

### 10.2 Storage trait

Ядро системи не пише у файли безпосередньо. Воно викликає методи trait'у `Storage`:

- `read_session(session_id) -> Session`
- `read_session_metadata(session_id) -> SessionMetadata`
- `write_session_metadata(metadata)`
- `append_event(session_id, event)`
- `read_session_archived_events(session_id) -> Vec<Event>`
- `rotate_session_events(session_id, covered_event_ids)`
- `write_archive_entry(entry)`
- `update_archive_entry(id, patch)`
- `read_archive(filters) -> StorageCollection<ArchiveEntry>`
- `read_memory_units_for_archive(archive_id) -> StorageCollection<MemoryUnit>`
- `read_core_store_category(category) -> CoreStoreCategory`
- `read_core_store_categories() -> StorageCollection<CoreStoreCategory>`
- `write_core_store_category(category)`
- `save_task(task)`, `load_tasks() -> StorageCollection<PendingTask>`
- `write_sleep_run(run)`, `load_sleep_runs() -> StorageCollection<SleepRun>`
- `begin_journaled_operation(op_id)`, `complete_journaled_operation(op_id)`

Колекційні читання толерантні: один пошкоджений JSON-файл пропускається і повертається як warning у `StorageCollection`, щоб recall/context/debug могли деградувати мʼяко. Point-read за конкретним id лишається strict і має повертати помилку.

Дефолтна імплементація `FileStorage` працює з файловою розкладкою вище. Інші імплементації (`IndexedStorage`, in-memory для тестів) реалізують той самий trait. Це дає тестам швидке in-memory сховище, а майбутньому індексу — місце.

### 10.3 Crash safety

Implementation note 2026-06-10: runtime sleep recovery is currently provided by
durable `SleepRun` checkpoints in `memory/runs/sleep/` plus idempotent
event-coverage rules. The `memory/journal/` primitive exists in `Storage`, but
v0.2 runtime sleep does not yet use it as a multi-file transaction journal.
Journal-based recovery is deferred until schema migrations, raw-event
compaction, or another operation needs explicit recover/rollback semantics.

**Один файл:** запис іде у `<file>.tmp`, виконується fsync, потім `rename(<file>.tmp, <file>)`. На більшості файлових систем це атомарна операція.

**Мульти-файлові операції, коли вони будуть підключені до journal:** перед операцією у `memory/journal/<op_id>.json` пишеться запис із описом задуму (які файли зміняться, який цільовий стан). Після успішного завершення операції journal-запис помічається `completed: true` і його можна видалити. На v0.2 це опис цільового механізму для майбутніх міграцій/recovery-heavy операцій; runtime sleep зараз покладається на durable `SleepRun` checkpoints і idempotent event coverage. Коли journal вмикається для конкретної операції, при запуску ядро системи перевіряє `memory/journal/` і:

- завершує операції з повним описом задуму, якщо це безпечно (наприклад, sleep можна повторити);
- відкочує операції, які не можуть бути безпечно завершені;
- лишає ті, що потребують ручного огляду власника.

### 10.4 Manifest

`memory/manifest.json` містить:

- `engine_version` — версія ядра системи, що останньою писала у це сховище;
- `schema_versions` — окремо для шарів Сесія / Архів / шар Ядро / Tasks / Journal;
- `active_embedding_model_id` — порожнє на v0.1;
- `last_migration_at` — час останньої успішної міграції;
- `created_at`, `updated_at`.

Manifest читається при кожному запуску ядра системи. Якщо схеми у файлах нижчі за схеми, що їх підтримує поточне ядро — запускається міграція. Якщо вищі — ядро відмовляється стартувати з повідомленням про несумісність (хост має оновитись).

## 11. Версіонування і сумісність

Усі версії — semver-подібні (`MAJOR.MINOR.PATCH`), але без жорсткого semver-контракту, поки проєкт у pre-1.0.

**engine_version** — версія Rust-крейту ядра системи. Змінюється при будь-якій зміні публічного API або поведінки.

**schema_version_session**, **schema_version_archive**, **schema_version_core**, **schema_version_task**, **schema_version_journal** — окремі версії для кожної персистентної структури. Зміна формату одного шару не вимагає міграції решти.

**prompt_version** — кожен файл у `prompts/` має у frontmatter поле `version`. Зміна промпта sleep-компресії може змінити якість архіву, тому це окрема одиниця змін з обов'язковим записом у `HISTORY.md`.

**embedding_model_id** — рядковий ідентифікатор моделі, що embed'ила архівний спогад (наприклад, `openai-text-embedding-3-small@v1`). Порожнє на v0.1. Дозволяє виявити mismatch і запустити re-embed-міграцію.

**adapter_version** — кожен адаптер декларує свою версію і діапазон сумісних версій ядра. Якщо діапазони не перетинаються, адаптер відмовляється підключатись.

**Політика міграцій.** Будь-яка зміна schema_version_* у бік збільшення повинна супроводжуватись кодом міграції, що читає старий формат і пише новий. Міграції виконуються при запуску, якщо manifest показує меншу версію. Міграції журналяться у `memory/journal/`, як і будь-яка інша мульти-файлова операція.

## 12. Промпти

Промпти живуть тільки у `prompts/`. Жоден промпт не вбудований у Rust-код.

**Структура промпт-файлу.** Простий текстовий файл `<prompt_id>.md` із YAML-frontmatter'ом:

```
---
prompt_id: sleep_compression
version: 1
description: Compresses one session into one archive memory.
inputs:
  - session_summary: text
  - events: list of structured events
  - hints: optional context hints
output_schema: sleep_compression_result_v1
role_hint: balanced
---

<тіло промпта тут>
```

**Як промпти виконуються.** Ядро системи не знає `prompts_dir` і не читає prompt-файли. Воно створює `LlmRequest` / durable task із `prompt_id`, `prompt_version`, `role_hint`, `prompt_inputs` і очікуваною schema/formою відповіді. Хост при виконанні задачі сам читає файл `<prompts_dir>/<prompt_id>.md`, підставляє inputs, вибирає provider/model/key і викликає свого LLM-провайдера. Потім хост повертає в ядро тільки `LlmResponse` (`text` або structured error). Це зберігає provider boundary: orchestration живе в ядрі, prompt loading і мережевий I/O — в хості.

**Принцип "не вигадувати наперед" зі стратегії.** У `prompts/` живуть тільки ті файли, які реально викликає код: Telegram chat system prompt, sleep/memory-unit/fidelity/reflection/forgetting prompts та інші активні LLM tasks. Нові промпти створюються тільки тоді, коли відповідна функція реально їх використовує.

## 13. Конкурентний доступ

На v0.2 ядро має внутрішньопроцесну fine-grained модель конкурентності:

- публічні методи `MemoryEngine` працюють через `&self`, не через глобальний `&mut self`;
- PyO3 adapter відпускає GIL навколо важких Rust-викликів (`allow_threads`);
- `LockRegistry` серіалізує тільки конкретні ресурси: `session:<id>`, `core:<category>`, `candidate:<id>`, `manifest` та інші вузькі ключі;
- lock-ordering фіксований: коли потрібне вкладення, воно йде `session -> core`; Core-locks не тримаються кілька одночасно;
- різні session ids можуть оброблятися паралельно в одному процесі.

Ця гарантія **внутрішньопроцесна**. Два окремі процеси над одним `memory/` не синхронізовані через filesystem locks; operational rule для v0.2 — один writer-процес на один runtime memory directory. Якщо другий host/debug tool має працювати над тим самим сховищем одночасно, потрібен окремий міжпроцесний lock або server-mode wrapper.

## 14. Адаптери

### 14.1 Python-адаптер

Реалізується через PyO3 як sub-crate `memory_engine_python`. Експортує мінімум:

- `Engine.new(config_path) -> Engine`
- `engine.ingest(event_dict) -> IngestResult`
- `engine.recall(query_dict) -> list`
- `engine.core_context_package(request_dict) -> dict`
- `engine.render_memory_view(package_dict, current_user_message) -> str`
- sleep/reflection/fidelity/forgetting driver methods: `begin_*`, `next_*_batch`, `submit_*_response`, `finish_*`
- `engine.pending_tasks() -> list`
- review/admin methods: candidates, core patching, forgotten/remember_back, flush stats

`event_dict` і `query_dict` — звичайні Python-словники з полями, описаними у розділах 4 і 6. Адаптер конвертує їх у Rust-структури.

LLM виклики виконує **сам хост**, не Rust core. Адаптер передає `LlmRequest` у Python-код бота/хоста; хост рендерить prompt-файл, виконує задачу через свого Anthropic/OpenAI/Google клієнта і повертає результат через відповідний `submit_*` метод. Це чесна реалізація рішення з розділу 3.1: ядро володіє orchestration/state machine, але не знає провайдерів, ключів, моделей або мережі.

### 14.2 Godot-адаптер (v0.3 candidate)

GDExtension-крейт `memory_engine_godot`. Експортує клас `MemoryEngine` як Godot Node. Той самий набір методів, але з типами, природними для Godot (`Dictionary`, `Array`, `String`). Технічні нюанси (мобільні збірки, статичне лінкування) виявляться при реалізації, тому не фіксуємо їх тут.

## 15. Що в v0.1, що в v0.2, що відкладено

### v0.1 (foundation)

- Rust-крейт ядра системи з публічним API для ingest/recall/context/sleep.
- Storage trait + FileStorage імплементація.
- Шар Сесія: ingest, ліміти, попереднє стискання живого хвоста.
- Шар Архів: запис, читання, multi-track archive entry, базові ваги, decay.
- Шар Ядро: Core Store як читання-запис, Core Context Package збирання.
- Recall стадія 1 (фільтр + scoring).
- Pull-based sleep orchestration: ядро володіє graph/retry/fail-soft/fallback, хост виконує `LlmRequest -> text`.
- Basic rolling / partial sleep: старша частина великого unarchived window переходить в archive, молодший active tail лишається в `session_recent` / `session_trace`.
- PendingTask/SleepRun механіка: створення, серіалізація, resume, відновлення після обриву.
- Manifest, версіонування схем, базові міграції-stub.
- Crash safety: атомарний запис + durable SleepRun checkpoints; journal primitive є, але runtime sleep ще не покладається на нього як transaction journal.
- Fine-grained in-process locking і PyO3 `allow_threads`.
- Python-адаптер через PyO3.
- Інтеграція в Telegram-бот як перший хост.
- Multi-pass sleep: memory-unit/compact projection, emotional, topic thread, personal signal, relational, consolidator prose overlay; structure збирає ядро.
- Multi-track ArchiveEntry: `compact_memory`, `emotional_markers`, `topic_thread`, `personal_signals`, `relational_tone`.
- Реальні prompt-файли у `prompts/` тільки для активних функцій.
- Тести: unit/integration на scoring, recall, sleep, context, driver і adapter.

### v0.2 (living memory)

- Memory units як атомарна форма спогаду поверх archive entries: коротка теза, джерела, вага, status, fidelity/review metadata.
- Evidence pack builder для малих source-grounded перевірок.
- Fidelity validator для перевірки, що стиснений спогад не викривив raw events або source archive.
- Reflection-процес і candidate beliefs, які агенти пропонують, але не промоутять у Core напряму.
- Promotion у Core через структурний lifecycle: candidate → ready_for_review → rejected/promoted → active CoreFact.
- Contested Core lifecycle: суперечливий approved candidate позначає старий active факт як `contested`, не перезаписує його мовчки.
- Forgetting як окремий агентний review: рутина позначається як `Forgotten` і виходить з default recall, але audit trail лишається; `/remember_back` повертає unit.
- Recall hygiene: time decay і buffered recall stats без перезапису archive файлів на кожен recall.
- Raw-event rotation: archived raw segments для подій, покритих Complete-архівами, без втрати evidence/audit доступу.
- Tolerant collection reads: один пошкоджений JSON у колекції не валить увесь recall/context package, warning виходить у notes.
- Formal end-to-end acceptance: `living_memory_cycle.rs`.

### v0.3+ (відкладено навмисно, не забуто)

- Godot-адаптер через GDExtension як другий host і справжній тест reusability ядра.
- Recall стадія 2 (embeddings) і стадія 3 (LLM re-rank) — після стабілізації Core/reflection/forgetting і тільки opt-in.
- Track-aware recall, що може діставати і ранжувати окремі `emotional_markers`, `topic_thread`, `personal_signals`, `relational_tone`, а не тільки весь archive entry.
- Reviewer-pass / core candidate reviewer, який дорогою моделлю формулює коротку щільну Core-тезу перед review.
- Автоматична промоція в шар Ядро (зняття обов'язкового ручного огляду) — тільки після того, як побачимо реальну якість candidates на ручному режимі.
- Unit-level recall counters замість archive-level recall proxy у forgetting protection.
- Per-scope Core store layout, якщо `core:<category>` стане bottleneck.
- Інтеграція з третім проєктом (специфікація буде, коли проєкт буде названий).
- Розширений набір LLM task-типів: recall_rerank, compute_embedding, fact_check, tag_proposal.
- Індексована імплементація `Storage` (sled або sqlite над файлами).

## 16. Що цей документ навмисно не задає

- **Точні JSON-схеми** подій, архівних спогадів, Core-фактів і PendingTask-результатів. Це окремий документ "Контракти даних", який пишеться наступним.
- **Назви Rust-модулів, структур і функцій.** Це окремий документ "Структура крейту", що пишеться разом із початком коду.
- **Точні формули scoring, decay, recall ranking.** Базові форми описані словами; конкретні коефіцієнти і функції підбираються емпірично і документуються окремо.
- **Тіло промптів.** Промпти створюються тоді, коли відповідна функція їх використовує (принцип зі стратегії, оновлення 2026-05-17).
- **Конкретний вибір LLM-провайдерів, моделей, API-ключів.** Це повністю поза ядром системи — у конфігу хоста.

## Підсумок

Memory Engine v0.2 — це Rust-бібліотека (ядро системи) з трьома шарами пам'яті (Сесія, Архів, шар Ядро), що приймає нормалізовані події від хоста, повертає компактний prompt-facing memory view, виконує pull-based LLM orchestration без provider/network залежностей у ядрі, і підтримує повний цикл живої памʼяті: sleep → memory units → recall/decay → fidelity → reflection candidates → manual review → Core/contested → forgetting/remember_back.

Перший хост — Telegram-бот через PyO3. Після v0.2 хост лишається тонким: він збирає domain state, виконує `LlmRequest` через свого provider-а, показує команди/debug і телеметрію. Логіка памʼяті — scoring, sleep graph, fail-soft, Core bridge, fidelity, reflection, contested lifecycle, forgetting protection і prompt projection — належить Rust-ядру. Наступна велика перевірка reusability — другий host, ймовірно Godot-адаптер.
