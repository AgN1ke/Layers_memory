# v0.1 Acceptance Checklist

Цей документ фіксує межу між "foundation зібраний" і "v0.1 підтверджений у живому використанні".

## Правила

- Перевірка виконується на реальному Telegram Gemini bot host.
- Runtime дані можна дивитися в `hosts/telegram_gemini_bot/runtime/`.
- Секрети не записуються у файли.
- Якщо пункт не проходить, не розширювати scope до v0.2. Спочатку виправити цей пункт.
- Після успішної перевірки додати короткий запис у `DEVLOG.md` з transcript summary, файлами runtime і результатами.

## Перед Стартом

- [ ] Запущено `.\hosts\telegram_gemini_bot\run_gui.ps1`.
- [ ] Вставлено Telegram bot token і Gemini API key.
- [ ] `auto-sleep events` лишено `50` або свідомо зменшено для швидкого тесту.
- [ ] У `hosts/telegram_gemini_bot/runtime/logs/bot.log` є новий рядок старту bot-а.
- [ ] У системі запущена тільки одна копія `bot.py`.

## 1. Core Команда

- [ ] Явно додати потрібні тестові факти через `/remember ...`, якщо runtime не містить scoped Core-фактів для цього чату.
- [ ] Надіслати `/core`.
- [ ] Bot показує тільки факти, які були збережені для цього chat scope через explicit `/remember` або через Archive → Core bridge після completed sleep, наприклад:
  - `Користувача звати Микита.`
  - `Користувачу 36 років.`
  - `Бота звати Аврора.`
  - `Користувач хоче спілкуватися на ти.`
- [ ] Plain text на кшталт `Мені 36 років` не створює CoreFact напряму в момент повідомлення.
- [ ] Такий plain text отримує event-теги в `events.jsonl` (`age_reference`, `personal_fact_signal`), якщо heuristic tagging спрацював.
- [ ] Після `/sleep` або auto-sleep LLM `personal_signals` можуть пройти Archive → Core bridge, якщо мають достатню confidence, user-authored source event і не дублюють існуючий факт.
- [ ] `/core` показує `core_fact_id` для кожного факту.
- [ ] `/core_seed` повторно проходить completed archive entries і seed-ить Core тільки з `personal_signals`, не з raw text.
- [ ] `/core_update <id> <новий текст>` оновлює факт у поточному chat scope.
- [ ] `/core_forget <id>` позначає факт як `deprecated`, після чого він не повертається в `/core` і prompt-context.

## 2. Core Холодний Старт

- [ ] Зупинити bot.
- [ ] Запустити bot заново через GUI.
- [ ] Надіслати "Як мене звати?"
- [ ] Bot відповідає "Микита" на основі `core_facts`.
- [ ] Надіслати "Як тебе звати?"
- [ ] Bot відповідає "Аврора" на основі `core_facts`.
- [ ] Надіслати "Ми на ти?"
- [ ] Bot підтверджує стиль звертання на "ти".

## 3. Multi-Chat Core Isolation

- [ ] Написати bot-у з другого Telegram chat/account.
- [ ] До представлення другого користувача bot не називає його Микитою.
- [ ] У другому чаті явно зберегти факт через `/remember Користувача звати Аліса.`
- [ ] У другому чаті запитати "Як мене звати?"
- [ ] Bot відповідає "Аліса".
- [ ] У першому чаті `/core` далі показує Микиту, 36, Аврору і "на ти".
- [ ] У другому чаті `/core` показує Алісу і не показує факти Микити.

## 4. Auto-Sleep

- [ ] Провести довгу сесію з 50+ новими повідомленнями або зменшити поріг у GUI для тесту.
- [ ] У `bot.log` зʼявляється рядок `auto-sleep completed`.
- [ ] У `runtime/memory/archive/2026/05/` зʼявляється новий archive `.json`.
- [ ] Archive має `status: complete`, не тільки `preliminary`.
- [ ] Archive має human-readable `gist` і `narrative`.
- [ ] Відповідний task у `runtime/memory/tasks/` має `state: completed`.

## 5. Topic Recovery З Archive

- [ ] Після auto-sleep перейти на нову тему і написати ще 20+ повідомлень.
- [ ] Запитати про стару тему, яка вже мала потрапити в archive.
- [ ] Bot згадує стару тему.
- [ ] У `bot.log` немає помилок під час відповіді.
- [ ] Через `/recall <стара тема>` видно релевантний archive item.

## 5.1 Context Budget Benchmark

- [ ] Після довгої сесії context package для відповіді вкладається максимум у **11k токенів** памʼяті.
- [ ] Поточна памʼять займає до **7k токенів**: active session tail, current turn/domain context, живі незаархівовані події.
- [ ] Стиснута памʼять займає до **3k токенів**: `archive_relevant`, тобто сконсолідовані archive memories.
- [ ] Core займає до **1k токенів**: активні `core_facts` для поточного scope.
- [ ] Якщо доводиться скорочувати контекст, не випадають головні сенси: emotional markers, personal signals, важливі люди/тварини/місця, зміни стосунку, рішення і незакриті питання.
- [ ] Перевірка не зараховується, якщо бюджет досягнуто простим обрізанням рядків без правил значущості.

## 6. Prompt І Provider Boundary

- [ ] Активний чат-промпт читається з `prompts/telegram_chat_system.md`.
- [ ] Sleep-prompts читаються з `prompts/`: `sleep_emotional_pass.md`, `sleep_topic_thread_pass.md`, `sleep_personal_signal_pass.md`, `sleep_relational_pass.md`, `sleep_consolidator.md`.
- [ ] У Rust core немає Gemini/OpenAI/Anthropic model id.
- [ ] Model mapping змінюється в host/GUI/env, не в бібліотеці.

## 7. Runtime Readability

- [ ] Поточна сесія має читабельні `session.md` і `events.jsonl`.
- [ ] Core facts лежать у зрозумілих файлах `profile.json`, `preferences.json`, `relationship.json`.
- [ ] Archive entries лежать по датованих папках.
- [ ] `bot.log` достатній для діагностики: старт, update id, chat id, message id, model, auto-sleep.
