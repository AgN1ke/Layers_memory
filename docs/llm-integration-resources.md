# LLM Integration Resources

## Для чого існує цей документ

Цей документ пояснює, які ресурси має надати програма, що вбудовує Memory
Engine.

Memory Engine є бібліотекою пам'яті. Вона зберігає події, стискає розмову,
піднімає важливі факти, перевіряє пам'ять, шукає далекі спогади і збирає
контекст. Але сама бібліотека не володіє ключами до провайдерів і не викликає
Google, OpenAI, Anthropic, DeepSeek або інші API напряму.

Програма, яка вбудовує бібліотеку, має підвести до неї модельні ресурси.

## Мінімальний набір

Для найпростішого запуску достатньо одного текстового LLM.

Один і той самий LLM можна використати для всіх текстових ролей:

- сильна робота зі змістом;
- звичайна робота зі стисканням пам'яті;
- дешеві прості проходи.

Це підходить для локального тесту, демо або першого запуску продукту.

Приклад: усе працює через один Gemini, один GPT, один Claude або один DeepSeek.

## Рекомендований набір

Для нормального продуктового використання краще мати два або три текстові
ресурси.

Сильна модель:

- перевірка, чи стиснена пам'ять не збрехала;
- пошук суперечностей;
- рефлексія над старими спогадами;
- ремонт слабко збережених архівів.

Звичайна модель:

- щоденне стискання розмови;
- виділення коротких тез;
- виділення персональних сигналів;
- збір звичайної довгої пам'яті.

Дешева модель:

- прості службові проходи;
- задачі, де не потрібне глибоке міркування;
- економія вартості там, де якість сильної моделі не потрібна.

Ці ролі можна мапити на будь-яких провайдерів. Наприклад:

- сильна модель: Claude або DeepSeek;
- звичайна модель: GPT або Gemini;
- дешева модель: Gemini Flash, дешевий GPT або інша швидка модель.

Також допустимо лишити одну модель для всіх трьох ролей, якщо продукт хоче
простішу конфігурацію.

## Векторна пам'ять

Векторна пам'ять потребує окремого ресурсу для embeddings.

Рекомендований варіант для приватності: локальна embedding-модель у програмі,
яка вбудовує бібліотеку. Текст пам'яті тоді не відправляється у зовнішній
embedding API.

Якщо векторна пам'ять вимкнена, embedding-ресурс не потрібен.

Якщо векторна пам'ять увімкнена, програма має вміти:

- порахувати embedding для коротких тез пам'яті;
- порахувати embedding для поточного запиту користувача;
- передати готові вектори в Memory Engine.

Memory Engine зберігає і шукає вектори, але не рахує їх сам.

## Що саме має надати програма

Програма, яка вбудовує бібліотеку, має мати власний шар виконання LLM-запитів.

Цей шар повинен:

- знати, де лежать ключі;
- знати, який провайдер і яка модель відповідає кожній ролі;
- викликати API провайдера;
- повертати результат у форматі, який очікує Memory Engine;
- обробляти помилки провайдера і таймаути;
- логувати використання без запису секретів у git.

На практиці це означає таку інфраструктуру в продукті:

- config або UI налаштувань, де власник продукту вибирає провайдерів і моделі;
- безпечне місце для ключів, яке не потрапляє в git;
- код, який уміє виконати текстовий запит на вибраному провайдері;
- код, який приводить відповідь провайдера до формату Memory Engine;
- політика timeout/retry для збоїв провайдера;
- локальний embedding-виконавець, якщо увімкнена векторна пам'ять;
- runtime-логи без текстів секретів і без повних приватних дампів пам'яті.

Ключі можуть приходити з:

- environment variables;
- локального config-файлу, який ігнорується git;
- runtime-сховища застосунку;
- UI налаштувань продукту.

## Що отримує Memory Engine

Memory Engine отримує не ключі і не назву провайдера як головну істину, а
готовий результат виконання задачі.

Для текстових задач програма отримує від бібліотеки запит із роллю роботи,
наприклад сильна, звичайна або дешева. Програма виконує цей запит на вибраній
моделі і повертає текст або структурований результат.

Для embeddings програма повертає готові числові вектори.

Це дозволяє одному продукту використовувати Gemini, іншому GPT, третьому
Claude, четвертому DeepSeek, а пам'ять при цьому лишається тією самою.

## Приклади конфігурацій

Простий режим:

- усі текстові ролі: Gemini;
- вектори: вимкнені.

Продуктовий режим:

- сильна модель: Claude;
- звичайна модель: GPT;
- дешева модель: Gemini Flash;
- embeddings: локальна модель.

Економний режим:

- сильна модель: DeepSeek;
- звичайна модель: DeepSeek;
- дешева модель: дешевий GPT або Gemini Flash;
- embeddings: локальна модель.

Локальний приватний режим:

- текстова модель: локальний або self-hosted LLM, якщо він витримує потрібну
  якість;
- embeddings: локальна модель;
- зовнішні API: не використовуються.

## Поточний стан репозиторію

У репозиторії вже є робочі демо-застосунки з Gemini-виконанням. Це доводить
інтеграційний шлях, але не означає, що бібліотека прив'язана до Gemini.

Спільний шар провайдерів для OpenAI, Anthropic, DeepSeek, Kimi та інших
провайдерів ще треба оформити окремою роботою: єдиний інтерфейс виконання,
приклади config-файлів, тести і документація для кожного провайдера.

## Практичний висновок для розробника

Щоб вбудувати Memory Engine у свій продукт, розробник має підготувати:

- місце для пам'яті на диску;
- спосіб передавати події користувача в бібліотеку;
- LLM-виконавець для текстових ролей;
- ключі або локальні моделі для цих ролей;
- embedding-виконавець, якщо потрібна векторна пам'ять;
- UI або config, де власник продукту вибирає провайдерів і моделі.

Після цього Memory Engine може виконувати свою частину: стискати, згадувати,
перевіряти, оновлювати стабільні факти і збирати короткий контекст для
відповіді.
