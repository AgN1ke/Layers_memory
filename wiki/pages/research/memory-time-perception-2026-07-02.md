# Відчуття часу в памʼяті: точний час свіжого, приблизний час старого (2026-07-02)

## Для чого існує цей документ

Цей документ фіксує постановку власника проєкту, аналіз поточного стану Memory Engine проти коду (`prompt_view.rs`, `recall_stage1.rs`, `session_ops.rs`, контракти) і запропоноване рішення.

Документ адресований ШІ-агенту, який працює з кодом. Його задача: перенести рішення в `wiki/pages/planning/roadmap.md`, узгодити з `wiki/pages/foundation/architecture.md` і `wiki/pages/foundation/contracts.md`, і реалізувати однією малою гілкою. Зміна поведінки prompt view проходить через `HISTORY.md` за загальною дисципліною.

## Інваріанти, які не можна зламати

Стандартний список проєкту діє повністю. Для цієї роботи критичні:

1. Ядро системи не виконує жодних мережевих викликів. Це виключає будь-які «часові API» в ядрі за конструкцією.
2. Файли — джерело правди; усе похідне перебудовується.
3. Не вигадувати наперед: конфігурація зʼявляється тільки під реальний код.
4. Сумісність: нові поля — опціональні через serde default, без міграцій.
5. Storage/debug view повний; prompt view семантично достатній і компактний.

## Частина 1. Проблематика (постановка власника)

Бібліотека вже підключена до реального Godot-проєкту і працює. Наступна потреба — щоб памʼять мала відчуття часу:

1. Свіжа («основна») памʼять може нести точний час, типу `10:35 02.07.2026`.
2. Старі спогади мають ділитися на приблизні кроки: вчора, позавчора, N днів тому; якщо більше ~7 днів — «давно в цьому місяці» або «давно в минулому місяці».
3. Якщо бібліотека довго не працювала, при запуску вся ця інформація має стати актуальною.
4. Якщо в системі немає часу (хост не підʼєднав) — усе має працювати і без нього; але бажано, щоб час був.
5. Відкрите питання власника: звідки брати час — із системи чи з мережі по API. Вимоги: логічно і раціонально, без постійного спаму запитами, без підозрілого для фаєрволів трафіку.

## Частина 2. Що показав аналіз коду

### 2.1. Абсолютний час уже є всюди; у prompt його немає ніде

**Сховище вже повністю датоване.** `StoredEvent.timestamp` + `received_at`, `ArchiveEntry.time_range {start, end}` + `created_at` + `updated_at`, `MemoryUnit.created_at`, `CoreFact.created_at/updated_at`, `CoreContextPackage.created_at`. Жодних нових полів для зберігання часу не потрібно.

**Prompt view не рендерить час взагалі.** `render_memory_view` подає `<long_memory>` як `- [score] теза`, `<short_memory>` як `user:`/`assistant:` рядки — без жодної часової мітки. Модель зараз не має ані «зараз», ані віку спогадів: питання «коли ми це обговорювали?» принципово без відповіді, а «вчора/сьогодні» у відповідях бота — галюцинація.

**Патерн «час — це вхід, а не ambient» уже існує.** `RecallQuery.created_at` — опціональний інжектований час із fallback на системний UTC (`query.created_at.map_or_else(now_rfc3339, Ok)`); саме так детерміновані тести інжектять час у decay. Рішення нижче лише поширює цей самий патерн.

**Прецедент «обчислюй при запиті, не зберігай» уже закладений.** Effective freshness рахується в момент recall від `reference_at` (Запис 83), а не зберігається як значення, що гниє. Часові мітки — той самий клас.

### 2.2. Відповідь на пункт 3 власника: «оновлення при запуску» не потрібне взагалі

Це головний дизайн-висновок: **відносні мітки не зберігаються — вони є функцією `label(stored_time, now)`, обчисленою в момент рендеру prompt.** Збережена мітка «вчора» завтра стає брехнею; обчислена — завжди актуальна. Тому після місяця простою бібліотеці нічого оновлювати: перший же `core_context_package` порахує мітки від поточного «зараз». Пункт 3 розчиняється в правильній структурі рішення, окремого «startup refresh pass» не існує.

## Частина 3. Рішення

### 3.0. Принцип

Час — метадані рендеру, не зміст і не стан. Сховище лишається як є (абсолютний UTC), prompt отримує похідні мітки, розраховані на момент збірки пакета. Жодних нових网络-залежностей, жодних збережених відносних значень.

### 3.1. Що саме зʼявляється в prompt

1. **`<state>` отримує один рядок `current_time:`** — наприклад `current_time: 2026-07-02 10:35 Wednesday (UTC+3)`. Це точка відліку для моделі: усі відносні мітки нижче стають однозначними, а питання «яке сьогодні число / який день тижня» отримує відповідь. Один рядок токенів.
2. **`<long_memory>` (archive items): мітка віку на item.** Формат: `- [yesterday | 0.88] Іржа -> ...` (джерело — `RecallItem.time_range.end`, поле вже є; fallback — без мітки). 
3. **`<short_memory>`: денні розділювачі в older trace.** Перед групою рядків старішого хвоста — заголовок бакета (`[yesterday]`, `[2 days ago]`), один на групу, не на рядок — дешевше за токенами. Живий recent-хвіст — без міток (це «зараз»); `DialogueEvent` для цього отримує `timestamp` із `CoreContextEvent.timestamp` (поле вже є).
4. **Core-факти — без часу.** Шар Ядро позачасовий за визначенням; «факт із травня» — можлива пізніша ідея, зараз не робити (не вигадувати наперед).

### 3.2. Шкала бакетів

Календарні дні (не 24-годинні вікна — «вчора» це календарне поняття):

- сьогодні → `today HH:MM` (для діалогових рядків) / `earlier today` (для archive item);
- вчора → `yesterday`;
- 2–6 днів → `N days ago`;
- ≥7 днів, той самий календарний місяць → `earlier this month`;
- попередній календарний місяць → `last month`;
- далі — **обовʼязкове продовження драбини власника, інакше 3-місячний спогад брехатиме «last month»**: `N months ago`, від 12 місяців → `over a year ago`.

Мова міток — канонічна англійська в рендері ядра, узгоджено з існуючими структурними маркерами prompt view (`conversation_state: ongoing` тощо); модель природно вживає мову розмови у відповіді. Конфігурованість мови міток — тільки якщо live-тести покажуть проблему.

### 3.3. Часовий пояс

«Вчора» — локальне поняття, а сховище — UTC. Для Києва межа доби по UTC зсунута на 2–3 години; для розумної колонки («що я робила вчора?») це помітно. Рішення: опціональне поле `utc_offset_minutes` у `CoreContextRequest` (serde default = 0, тобто UTC) — хост знає свою локаль. Жодних таймзон-баз у ядрі: один цілочисельний offset.

### 3.4. Джерело часу — політика по шарах

**Ядро:** тільки injected time або системний UTC-годинник (`OffsetDateTime::now_utc()` — уже так). Мережа заборонена інваріантом. Це відповідь на половину питання 5: «часового API» в бібліотеці не буде за конструкцією.

**Хост — системний годинник за замовчуванням.** Усі поточні і заплановані хости (Windows, Linux, Android/Godot, колонка) мають системний годинник; реальний ризик не «часу немає», а «годинник бреше» (мертва CMOS-батарейка, embedded до NTP-синку).

**Мережева корекція — опортуністична, без жодного додаткового запиту.** Ключове спостереження: хости, які взагалі мають мережу, уже роблять HTTP-виклики (Gemini, Telegram API), і кожна HTTP-відповідь несе стандартний заголовок `Date` (RFC 9110). Хост може порівнювати системний час із `Date` відповіді, яку й так отримав: нуль нового трафіку, нічого підозрілого для фаєрволів, нема чому «спамити». Якщо дрейф перевищує поріг (наприклад, 5 хвилин) — хост передає скоригований «зараз» у запити до ядра або ставить прапорець недовіри (3.5). NTP-клієнт чи звертання до time-API — свідомо ні: зайва складність і саме той трафік, якого власник хоче уникнути.

**Офлайн-хост без мережі** (гра без інтернету, колонка до Wi-Fi): системний годинник — єдине джерело, і цього достатньо; якщо і його немає/він недовірений — режим деградації.

### 3.5. Деградація без надійного часу

1. Хост може позначити годинник недовіреним (конфіг/поле запиту). У цьому режимі часові мітки **опускаються** — рендер поводиться як сьогодні. Брехлива мітка «вчора» гірша за відсутню; решта памʼяті працює повністю.
2. Ядро саме детектить аномалію: якщо `now` менший за найновіший відомий timestamp сховища — мітки опускаються і в `package.notes` додається попередження (канал notes уже існує після A5).
3. Відʼємний вік уже неможливий: `timestamp_age_days` клемпить до `0.0`.

### 3.6. Точки в коді

1. Чиста функція(ї) бакетизації в ядрі + юніт-тести на межі: 23:59 вчора vs 00:01 сьогодні, зміна місяця, лютий, ненульовий offset, майбутній timestamp.
2. `prompt_view.rs`: рядок `current_time` у `<state>`; мітка в `render_archive_memory_prompt_lines`; денні заголовки в older trace. Budget лишається чесним автоматично — estimator і рендер діляться одними prompt-line функціями (властивість, зафіксована Записом 82).
3. `CoreContextRequest.utc_offset_minutes` (serde default). Час рендеру — `package.created_at` (уже є), нових входів у `render_memory_view` не потрібно.
4. HISTORY-запис: зміна поведінки prompt view. Юніт-тест prompt_view оновити; conformance-хости не ламаються (їхні asserts — маркери секцій і підрядки змісту), додати assert на мітку в одному сценарії.

### 3.7. Чого свідомо не робити

1. Не зберігати відносні мітки в жодному файлі.
2. Не вимагати від sleep-пасів запікати дати в тези memory units — тези позачасові, час додає рендер із `time_range`. (Якщо дата — сам зміст спогаду, «зустріч 5 липня», LLM природно лишить її в тезі; це не заборона.)
3. Не робити мережевих часових запитів ні в ядрі (заборонено), ні в хості (не потрібно — є `Date`-заголовки наявного трафіку).
4. Не тягнути таймзон-базу; один offset.
5. Не робити шкалу конфігурованою на старті — константи + тести; конфіг зʼявиться, якщо реальний хост попросить іншу шкалу.

## Частина 4. Послідовність і місце в roadmap

1. **Одна мала гілка**: бакетизація + prompt_view + `utc_offset_minutes` + тести + HISTORY. Незалежна від multi-speaker geometry, але **обидві правлять `prompt_view.rs` — робити послідовно, не паралельно** (порядок будь-який; часова гілка менша, її природно пустити першою).
2. Live-перевірка на Telegram host: «коли я тобі казав про X?» після кількох днів історії; на колонковому сценарії — «що я робила вчора?».
3. Хостова частина (опортуністична корекція за `Date`-заголовком) — окремий маленький пункт у Telegram/proxy host, не блокує гілку ядра.
4. Синергія на майбутнє: `current_time` у `<state>` потрібен і мультиспікерному чату (реплаї «на вчорашнє»), і колонці — це спільний фундамент обох напрямів.

## Частина 5. Критерії перевірки

1. **Сумісність:** старі дані читаються без міграції; запит без `utc_offset_minutes` працює (UTC); conformance-сценарії всіх хостів проходять без правок очікувань (крім навмисно доданого assert на мітку).
2. **Рендер:** пакет із учорашнім archive item дає `[yesterday | ...]`; той самий пакет, відрендерений через тиждень (інжектований `created_at`), дає `[7 days ago]`/`[earlier this month]` — детерміновано, без «оновлення» файлів.
3. **Простій:** сховище, що не використовувалося N днів, після відкриття одразу дає актуальні мітки — жодного startup-pass.
4. **Деградація:** прапорець недовіреного годинника → рендер без міток, байтово ідентичний сьогоднішньому; `now` у минулому відносно сховища → мітки опущені + note.
5. **Межі бакетів:** юніт-тести з 3.6.1 зелені, включно з offset.

## Підсумок

Сховище вже знає точний час усього; бракує лише перекладу цього часу в prompt. Рішення — три локальні зміни рендеру (current_time у state, мітка віку на archive item, денні розділювачі старого хвоста) плюс одна чиста функція бакетизації і один опціональний offset у контракті запиту. Відносний час ніколи не зберігається — тому «оновлення після простою» не існує як задача. Час береться з системного годинника; мережева корекція — тільки опортуністична, за `Date`-заголовками трафіку, який хост і так робить; без надійного годинника мітки чесно зникають, а памʼять працює далі.
