# Стратегія окремого Memory Engine на Rust

## Для чого існує цей документ

Цей документ описує стратегічний задум окремого проєкту Memory Engine. Його задача - зафіксувати намір, мотивацію, форму продукту і межі майбутньої розробки так, щоб на цій основі можна було створювати конкретні ТЗ, плани, контракти і задачі для коду.

Документ адресований ШІ-архітектору або розробнику, який буде проєктувати окремий модуль пам'яті. Перед плануванням реалізації він має зрозуміти, яку проблему вирішує Memory Engine, чому центральною формою обрано Rust-бібліотеку і як ця бібліотека має підключатися до різних застосунків.

Головна ідея проста: тришарова пам'ять уже стала окремою концепцією, яка використовується в Chibigochi, Telegram-боті та ще одному проєкті. У кожному місці вона зараз реалізована локально і тісно прив'язана до конкретного коду. Через це одна архітектурна ідея розходиться на різні версії.

Memory Engine має стати окремим спадковим модулем: власний Git-репозиторій, власна історія розвитку, власні контракти, власні тести і тонкі інтеграційні шари для різних застосунків.

## Намір

Мета Memory Engine - зробити тришарову пам'ять повторно використовуваною системою у формі реальної бібліотеки. Вона має брати на себе логіку пам'яті і давати різним проєктам один спільний механізм.

У Chibigochi така пам'ять потрібна для героїні, яка роками зберігає характер, пам'ятає важливе, забуває рутину і переживає ігрові події як власний досвід. У Telegram-боті та інших проєктах ця сама ідея потрібна для довготривалого контексту, стабільних фактів, історії взаємодії, накопичення висновків і контрольованого забування.

Спільною для всіх цих випадків є логіка пам'яті:

- є стабільне ядро;
- є поточна робоча сесія;
- є довгостроковий архів;
- є відбір важливого;
- є ваги спогадів;
- є стискання сесій;
- є поступове формування переконань;
- є забування рутини;
- є контроль лімітів;
- є потреба в надійному збереженні.

Саме ця спільна логіка має стати окремим Memory Engine.

## Проблема, яку потрібно вирішити

Зараз пам'ять існує як ідея, що повторюється в різних проєктах. Кожна реалізація має власні локальні рішення: формати, правила ваг, місця збереження, промпти, способи стискання і винятки. Навіть якщо всі реалізації почалися з однієї концепції, з часом вони розходяться.

Це створює кілька практичних ризиків.

Перший ризик - дублювання роботи. Коли треба покращити sleep-компресію, recall, формат архівного спогаду або міграцію старих файлів, це доводиться робити в кожному проєкті окремо.

Другий ризик - розпад архітектури. Один проєкт може зберегти суворе розділення Core, Session і Archive, інший почне змішувати їх, третій залишить стару схему, четвертий додасть швидке рішення під конкретний баг. Після цього вже складно сказати, де справжня версія пам'яті.

Третій ризик - втрата якості при масштабуванні. Поки проєкт один, ручна інтеграція виглядає терпимою. Коли проєктів кілька, пам'ять стає окремим продуктом фактично, але ще не оформлена як окремий продукт технічно.

Четвертий ризик - залежність від конкретного середовища. Якщо логіка пам'яті живе всередині Godot, вона погано переноситься в Python. Якщо вона живе всередині Telegram-бота, її важко нормально використати в грі. Якщо вона живе в промптах, її складно тестувати, версіонувати і захищати від випадкових змін.

Memory Engine потрібен для того, щоб ця логіка мала одне місце розвитку.

## Форма продукту

Memory Engine має існувати як окремий Rust core, який підключається до хост-проєктів як бібліотека або модуль. Хост-проєктами можуть бути Chibigochi, Telegram-бот та інші застосунки.

Rust core містить логіку пам'яті. Godot, Python та майбутні середовища отримують тонкі адаптери, які перекладають події конкретного застосунку в контракти Memory Engine і повертають назад готовий контекст, спогади та результати обробки.

Головна форма взаємодії така:

- хост-проєкт передає події, стани, повідомлення і завершення сесій;
- Memory Engine обробляє їх через Core, Session, Archive, sleep, recall і reflection;
- хост-проєкт отримує релевантний контекст, архівні спогади, стабільні факти, оновлені ваги і службові результати.

Це модульна архітектура для вбудовування в застосунки. Логіка пам'яті розвивається в одному ядрі, а конкретні продукти підключають її через свої адаптери.

## Стратегічний принцип

Memory Engine треба мислити як внутрішню технологію вищого рівня, ніж окремий клас у конкретному застосунку. Він має бути спільним ядром для різних продуктів, а не локальною реалізацією всередині кожного з них.

Кожна інтеграція адаптує свій світ до контрактів Memory Engine. Chibigochi, Telegram-бот і майбутні застосунки можуть відчувати пам'ять як природну частину себе, а логіка пам'яті при цьому розвивається окремо і послідовно.

Цей принцип вимагає архітектурної незалежності: власний репозиторій, документація, історія змін, версії, тести, схеми даних, адаптери, міграції і політика сумісності.

## Що має зрозуміти наступна ШІ

Наступна ШІ, яка буде працювати з цим документом, спочатку має зрозуміти задум.

Memory Engine народжується з конкретної проблеми: одна й та сама тришарова пам'ять уже потрібна в кількох проєктах, але щоразу реалізується заново. Завдання нового проєкту - перетворити цю пам'ять на спільне ядро, яке має власне життя, власну історію і власний розвиток.

Будь-яке майбутнє ТЗ має виростати з цього наміру: спільне ядро пам'яті, повторне використання в різних проєктах, контрольована інтеграція через адаптери, стабільні контракти і власна історія розвитку.

Стратегічний центр цього проєкту - окремий Memory Engine, який бере на себе логіку довготривалої пам'яті і дозволяє різним продуктам користуватися спільною системою.

## Rust-бібліотека як форма існування Memory Engine

Memory Engine має бути реалізований як окрема Rust-бібліотека. Це ключова архітектурна ідея всієї стратегії.

У цьому задумі Rust визначає форму продукту. Пам'ять має стати ядром, яке компілюється, тестується, версіонується і підключається до різних середовищ. Саме бібліотека дає потрібну модель: логіка пам'яті живе в одному місці, а конкретні програми використовують її через інтеграційні оболонки.

Rust core має бути центром відповідальності за пам'ять. У ньому живуть правила, які роблять систему тришаровою пам'яттю.

До відповідальності Rust core належать стратегічні механізми:

- прийом нормалізованих подій від зовнішніх проєктів;
- ведення робочої сесії;
- контроль лімітів сесії;
- оцінка важливості подій і система ваг;
- decay і зміна актуальності;
- правила переходу інформації між шарами;
- sleep-компресія;
- формування архівних спогадів;
- recall релевантних спогадів;
- підготовка candidate beliefs;
- промоція стабільних висновків у Core;
- перевірка схем, міграції і контроль цілісності даних.

Це ядро має бути самодостатнім настільки, щоб його можна було запускати в тестах без Chibigochi, Telegram-бота або іншого хост-проєкту. Окрема перевірка робить пам'ять самостійним продуктом.

## Чому саме Rust

Rust підходить для цієї ролі через поєднання кількох властивостей.

Перша властивість - надійність роботи з даними. Memory Engine буде працювати з довготривалими файлами, версіями схем, міграціями, індексами, журналами запису, архівами і станами. Помилки в такому шарі накопичуються роками, тому ядро має бути суворим до типів, меж форматів і станів збереження.

Друга властивість - зручність створення бібліотеки з кількома зовнішніми входами. Rust core може мати одну внутрішню модель, а назовні давати різні способи підключення: Python binding для Telegram-бота, GDExtension або Godot addon для гри, діагностичні інструменти для розробника.

Третя властивість - контроль продуктивності. Пам'ять може рости роками. Навіть якщо перші версії працюватимуть з невеликими файлами, архітектура має витримувати майбутні обсяги: багато сесій, багато архівних спогадів, пошук за тегами, фільтрацію за часом, переоцінку ваг, індексацію і міграції.

Четверта властивість - хороша межа між ядром і зовнішнім світом. Rust дозволяє зробити стабільний шар, який не залежить від стилю конкретного застосунку. Godot має свої сцени, сигнали, ресурси і user://. Python-бот має свої handlers, повідомлення і файлову структуру. Інший проєкт матиме свою модель. Rust core при цьому зберігає одну логіку.

П'ята властивість - довгострокова підтримуваність. Memory Engine задуманий як спадковий проєкт, який буде розвиватися довше за одну гру або один експеримент. Для такого проєкту важливо мати чіткі типи, тести, релізи, версії схем і контроль сумісності.

## Як Rust core має виглядати концептуально

Rust core потрібно мислити як внутрішній двигун пам'яті.

Зовнішній застосунок передає в нього структуровані факти про те, що сталося. Ядро приймає ці факти, звіряє їх із поточним станом пам'яті, вирішує, як вони впливають на Session, Archive і Core, оновлює внутрішні структури і повертає результат, який застосунок може використати.

Для Chibigochi це означає чіткий поділ: Godot відповідає за гру, сцени, стани героїні, UI, рейди, предмети і виклик агента. Memory Engine відповідає за пам'ять: сировину сесії, ваги, архів, candidate beliefs і релевантні спогади.

Для Telegram-бота Python-код передає повідомлення, відповіді, завершені сесії, ручні нотатки і важливі події в Memory Engine. У відповідь він отримує контекст для наступного запиту моделі або готові результати обробки.

Для третього проєкту це означає, що пам'ять підключається через той самий принцип: проєкт описує події своєю мовою, адаптер приводить їх до контрактів Memory Engine, ядро обробляє їх за спільними правилами.

## Словник термінів

`Rust core` - центральна бібліотека, де живе логіка пам'яті.

`Host project` - застосунок, який підключає Memory Engine: Chibigochi, Telegram-бот або інший продукт.

`Adapter` - тонкий шар між хост-проєктом і Rust core. Він перекладає події конкретного застосунку в контракти Memory Engine і повертає результат назад.

`LLM adapter` - окремий шар для мовних задач: compression, reflection, quote extraction, tag proposal, fact-checking assist.

`Core Store` - стабільні факти і висновки, які належать довготривалій основі пам'яті.

`Core Context Package` - готовий контекст для агента або LLM, зібраний із Core Store, поточного стану хост-проєкту і релевантних спогадів.

## Бібліотека і адаптери

Memory Engine має складатися з ядра і адаптерів.

Ядро - це Rust library. Воно містить правила пам'яті, моделі даних, міграції, збереження, recall і перевірки.

Адаптер - це тонкий шар між ядром і конкретним середовищем. Його задача - перекласти мову застосунку на мову Memory Engine і повернути результат назад у форматі, який цьому застосунку зручно використати.

Для Godot адаптер приймає ігрові події, стани персонажа, завершення дня, sleep-тригери і діалогові фрагменти. Він дає грі готові дані для AgentGateway: актуальний Core-контекст, релевантні архівні спогади, короткий стан сесії і результати консолідації.

Для Python адаптер працює з повідомленнями, чат-сесіями, командами, ручними нотатками і задачами. Він дає боту контекст для відповіді, результати збереження, витягнуті спогади і сигнали для sleep або reflection.

Для майбутніх проєктів адаптер має бути повторенням того самого принципу: мінімальний переклад між доменом застосунку і стабільним ядром пам'яті.

## Що має залишатися стабільним

Найважливіша частина Rust-ідеї - стабільна межа між ядром і споживачами.

Chibigochi може змінити UI, механіку рейдів, баланс станів, формат предметів або поведінку AgentGateway. Telegram-бот може змінити провайдера LLM, формат команд або спосіб запуску. Інший проєкт може мати зовсім інший домен. Memory Engine при цьому має зберігати стабільні контракти: як передати подію, як завершити сесію, як отримати recall, як оновити Core, як перевірити схему, як виконати міграцію.

Це дає головну практичну перевагу: майбутнє покращення пам'яті відбувається в одному проєкті. Коли змінюється алгоритм ваг, це стає покращенням Memory Engine. Коли з'являється нова стратегія decay, вона проходить через версію ядра. Коли покращується sleep-компресія, вона стає доступною всім інтеграціям. Коли знаходиться помилка в міграції, вона виправляється в одному місці.

Rust core у цій стратегії є джерелом правди для логіки пам'яті. Інтеграції мають бути достатньо тонкими, щоб підключати пам'ять до життя конкретного застосунку, але достатньо простими, щоб не дублювати саму пам'ять усередині себе.

## Хто передає інформацію в Memory Engine

Memory Engine має жити на межі між подіями застосунку і довготривалою пам'яттю. Конкретний проєкт знає, що сталося у його світі. Memory Engine знає, як перетворити це на пам'ять.

Тому перший потік даних іде від хост-проєкту до Memory Engine. Хост-проєкт передає події, стани, повідомлення, завершені сесії, важливі ручні нотатки і системні тригери. Memory Engine приймає їх як сировину для Session, Archive і Core.

У Chibigochi такими джерелами є діалоговий шар, ігровий стан, рейди, інвентар, магазин, сон, lifecycle-події і ручні дії гравця. Для пам'яті важливий не клас, який створив подію, а зміст: що сталося, у якому стані була героїня, як відреагував гравець і чи має ця подія довготривалий сенс.

У Telegram-боті джерела інші, але принцип той самий.

Бот передає репліки користувача, відповіді асистента, межі сесії, зміни теми, явні команди пам'яті і фонові тригери для sleep або reflection.

В іншому проєкті джерелами можуть бути задачі, рішення, документи, нотатки, архітектурні правки, домовленості, виправлені помилки або важливі події взаємодії. Для Memory Engine це різні форми одного принципу: зовнішній світ повідомляє, що сталося.

## Що саме передається

Хост-проєкт передає нормалізовану подію: тип, джерело, час, короткий опис, доменні дані і контекст.

Для Chibigochi одна подія може означати: героїня повернулась із рейду, принесла золото, втратила здоров'я, спорядження пошкодилось, гравець її вилікував, після цього вона відповіла певною реплікою. Для гри це набір механік. Для пам'яті це потенційний досвід.

Для Telegram-бота одна подія може означати: користувач розповів важливий факт про себе, пізніше уточнив його, а потім попросив згадати це у новому контексті. Для бота це повідомлення. Для пам'яті це кандидат на довготривалий факт або архівний спогад.

Для робочого або творчого проєкту одна подія може означати: було прийнято архітектурне рішення, потім знайдено недолік, рішення переглянули, а HISTORY має зафіксувати зміну. Для застосунку це робочий процес. Для пам'яті це історія висновків і корекцій.

Memory Engine має отримувати достатньо контексту, щоб правильно оцінити подію:

- хто або яка система створила подію;
- коли вона сталася;
- до якої сесії вона належить;
- який поточний стан був поруч;
- які теги або доменні категорії відомі хосту;
- який текст, результат дії або числовий стан треба врахувати;
- який зв'язок із попередніми подіями вже відомий;
- який режим обробки очікується: негайний або через sleep.

Це стратегічний контракт, а не остаточна схема. Конкретні JSON-формати мають з'явитися в окремому технічному ТЗ. На рівні стратегії важливо зафіксувати напрям: Memory Engine приймає осмислені події, а не прив'язується до внутрішніх класів конкретного застосунку.

## Хто бере інформацію з Memory Engine

Другий потік даних іде від Memory Engine назад у хост-проєкт.

У Chibigochi головним споживачем буде AgentGateway. Саме він збирає контекст для LLM і має отримувати з пам'яті те, що потрібно для поточної відповіді героїні: стабільний Core-контекст, релевантні архівні спогади, короткий стан поточної сесії, важливі факти про гравця, емоційні наслідки попередніх подій і поточні candidate beliefs, якщо вони впливають на поведінку.

Діалогова система використовує пам'ять для відповіді. Система поведінки може використовувати її для проактивності. UI може показувати журнал дня, короткі спогади або debug-представлення. Save/Lifecycle-шар отримує статус збереження, версію схем і результати міграцій.

У Telegram-боті головними споживачами будуть prompt assembler, conversation manager і command layer. Вони беруть релевантний контекст, стан сесії, рішення про sleep і результати команд пам'яті.

В іншому проєкті споживачами можуть бути модуль планування, модуль прийняття рішень, інтерфейс нотаток, агент, який пише відповіді, або система історії проєкту.

## Центральна роль recall

Recall - це один із головних способів, яким Memory Engine повертає цінність хост-проєкту.

Хост-проєкт ставить запит: для цієї ситуації, цього користувача, цього стану і цієї теми потрібен релевантний контекст. Memory Engine шукає у Core, Session і Archive те, що має сенс повернути. Результат має бути компактним, пояснюваним і придатним для вставки в LLM-контекст або для прямого використання логікою застосунку.

Для Chibigochi recall може повернути спогад про попередній складний рейд, факт про улюблену тему гравця, емоційний висновок із минулої розмови або Core-факт, який має вплинути на відповідь.

Для Telegram-бота recall може повернути довготривалий факт про користувача, попереднє рішення, незавершену тему або короткий підсумок старої сесії.

Для робочого проєкту recall може повернути причину архітектурного рішення, історію виправлення, попереднє застереження або зміну benchmark-claim.

Recall має бути контрольованим. Він має давати не просто схожий текст, а пам'ять, яка має вагу, джерело, час, тему і пояснення релевантності. Це потрібно для довіри: хост-проєкт має розуміти, чому саме ці спогади повернулися в контекст.

## Потік sleep і reflection

Окремими потоками є sleep і reflection.

Sleep запускається тоді, коли хост-проєкт повідомляє Memory Engine про завершення сесії, дня, діалогу або важливого циклу. У Chibigochi таким тригером може бути сон героїні, завершення дня, повернення з рейду або мобільний lifecycle-момент. У Telegram-боті це може бути завершення діалогу, довга пауза або ручний тригер. В інших проєктах це може бути завершення задачі, дня роботи або обговорення.

Під час sleep Memory Engine бере робочу сесію, оцінює значущість подій, відбирає важливе, стискає матеріал і формує архівні спогади. Хост-проєкт отримує оновлений Archive, стислу або очищену Session і статус обробки.

Reflection працює на довшій дистанції. Вона аналізує накопичені архівні спогади, шукає повторювані теми, суперечності, стабільні патерни і потенційні candidate beliefs. Хост-проєкт може отримати сигнал, що певний висновок дозрів і його можна розглянути для Core.

Ці потоки важливі для всіх інтеграцій. Вони роблять пам'ять живою системою, яка не просто накопичує записи, а поступово перетворює досвід на структуру.

## Межа відповідальності

Хост-проєкт відповідає за свій світ. Він знає, що таке рейд, магазин, повідомлення, задача, команда, предмет, стан здоров'я, день гри або діалог.

Memory Engine відповідає за пам'ять. Він знає, як прийняти подію, оцінити її значущість, зберегти її в правильному шарі, знайти її пізніше, стиснути сесію, оновити ваги, виконати міграцію і захистити дані.

Саме така межа дозволяє зробити систему універсальною. Chibigochi не втрачає свою ігрову специфіку. Telegram-бот не втрачає свою розмовну природу. Інший проєкт не змушений ставати схожим на гру. Усі вони користуються одним Memory Engine, бо передають йому події і беруть назад пам'ять.

## Внутрішня модель пам'яті

Memory Engine має зберігати початкову трирівневу ідею як свою внутрішню архітектуру. Ця модель є серцем проєкту:

- Core;
- Session;
- Archive.

Ці три шари мають різну роль, різний життєвий цикл, різний формат і різні правила зміни. Саме розділення шарів дозволяє пам'яті працювати роками, зберігати сталість особистості, не переповнювати контекст і не перетворювати довгу історію на некерований лог.

## Core

Core - це стабільна основа пам'яті.

У Chibigochi Core відповідає за те, що героїня пам'ятає як частину себе: ім'я, базовий сеттинг, стосунок до гравця, фундаментальні факти, найважливіші висновки, синтезовані переконання, сталі межі особистості. Поруч із цим у runtime-контексті можуть бути поточні стани з Godot: фізичний стан, емоційний стан, одяг, локація і поточний режим.

Для Memory Engine важливо розрізняти два типи Core-даних.

Перший тип - стабільний Core Store. Це те, що справді належить довготривалій пам'яті: фундаментальні факти, синтезовані переконання, сталі висновки, важливі дані про користувача або персонажа.

Другий тип - Core Context Package. Це готовий пакет для LLM або агента, який може містити стабільний Core Store разом із поточним станом, отриманим від хост-проєкту. У Chibigochi поточний стан героїні приходить із Godot. У Telegram-боті таким станом може бути активна тема, режим діалогу або профіль користувача. Memory Engine має допомагати зібрати цей пакет, але джерело поточного доменного стану залишається в хост-проєкті.

Core змінюється повільно. Новий факт або переконання має потрапляти в Core через контрольований процес. Для цього потрібні candidate beliefs: попередні висновки, які виникли з кількох архівних спогадів, але ще потребують підтвердження або чітких правил промоції.

Це дає пам'яті сталість. Персонаж або агент поступово розвивається, але не переписує свою основу після кожної окремої репліки.

## Session

Session - це робоча пам'ять поточної взаємодії.

У Chibigochi Session містить живий день або поточний ігровий цикл: діалоги, дії гравця, реакції героїні, зміни станів, контекст рейду, покупки, догляд, сон і короткі емоційні моменти. У Telegram-боті Session містить поточну розмову. В іншому проєкті Session може бути поточним обговоренням, задачею, робочою сесією або документом.

Session має бути зручною для вставки в LLM-контекст. Тому стратегічно правильний формат для цього шару - Markdown або plain text із мінімальними службовими маркерами. Цей шар має бути читабельним, компактним і придатним для стиснення.

Session має контрольований розмір. Пам'ять має вміти працювати з лімітами контексту, витісняти старіші частини, зберігати коротке резюме і продовжувати поточну взаємодію. Якщо сесія довга, старі події проходять попереднє стискання або потрапляють у буфер для sleep. Живий хвіст сесії залишається поруч із агентом, бо саме він підтримує відчуття безперервності.

Для надійності Session має підтримувати ранню фіксацію важливих подій. Подія з потенційною довготривалою цінністю записується до складної відповіді, довгої дії або ризикованого lifecycle-моменту.

Session є робочим простором, де події змагаються за майбутнє. Частина залишиться тільки в короткому контексті. Частина стане архівним спогадом. Частина після повторення або емоційної ваги перетвориться на матеріал для Core.

## Archive

Archive - це довгострокова пам'ять.

У цьому шарі зберігаються стиснуті спогади, які пройшли відбір. Archive має бути структурованим, бо з ним працює програма: його треба фільтрувати, індексувати, мігрувати, перевіряти, зберігати, відновлювати і використовувати для recall.

Стратегічно правильний формат Archive - JSON-орієнтована структура зі схемами і версіями. Конкретні поля мають бути визначені в технічних контрактах. На рівні стратегії важливо зафіксувати, що архівний спогад має нести час, тему, джерело, короткий зміст, емоційний наратив, факти, цитати, теги, вагу, актуальність, зв'язки і версію схеми.

Archive має зберігати не сухий протокол, а стиснений досвід. Для Chibigochi це критично: пам'ять героїні має відчуватися як пережитий день, а не як список технічних подій. Якщо гравець лікував її після важкого рейду, архівний спогад має зберегти не лише факт лікування, а й емоційний сенс цієї сцени. Якщо в розмові була яскрава фраза, коротка цитата може зберегти голос моменту.

Для Telegram-бота Archive може зберігати важливі факти про користувача, підсумки тем, рішення, повторювані мотиви і зміни позиції. Для робочого проєкту Archive може зберігати рішення, причини змін, виправлені помилки і контекст важливих висновків.

Archive живе довго, але його вміст має змінювати актуальність. Спогади мають вагу, decay, повторні звернення і зв'язки. Рутинні або давно неактуальні спогади можуть стискатися сильніше, втрачати пріоритет у recall або поступово виходити з активної зони. Ультраважливі спогади можуть отримувати захист через вагу, зв'язки або промоцію в Core.

## Ваги і природний відбір

Система ваг є механізмом, який перетворює потік подій на пам'ять.

Кожна подія або фрагмент сесії має отримати оцінку значущості. Значущість може залежати від емоційної сили, новизни, повторюваності, зв'язку з Core, впливу на стан, прямого прохання користувача, рідкісності, наслідків для майбутньої поведінки або зв'язку з важливою темою.

Для Chibigochi високої ваги можуть набирати події турботи, небезпечні рейди, сильні зміни стану, подарунки, конфлікти, важливі розмови, повторювані теми гравця, моменти відновлення після поганого стану. Низьку вагу можуть мати звичайні повторювані дії, які не створюють нового досвіду.

Для Telegram-бота високої ваги можуть набирати сталі факти про користувача, довготривалі цілі, зміна переваг, рішення, які впливають на майбутні відповіді, і явні прохання щось запам'ятати.

Вага не є одноразовою оцінкою. Спогад може ставати важливішим через повторні звернення, зв'язки з новими подіями або участь у формуванні Core-висновку. Спогад може втрачати актуальність через час, відсутність звернень або заміну новішим знанням.

Ця система потрібна для головної властивості пам'яті: вона має зберігати важливе і поступово відпускати рутину.

## Sleep

Sleep - це процес консолідації сесії.

У Chibigochi sleep природно пов'язаний із сном героїні, завершенням дня, поверненням із рейду або lifecycle-подіями мобільної гри. У боті sleep може бути завершенням діалогу або фоновим процесом після довгої паузи. В інших проєктах sleep може означати завершення робочої сесії.

Під час sleep Memory Engine бере Session і перетворює її на довготривалу структуру. Він виділяє важливі події, стискає матеріал, формує архівні спогади, коригує ваги, створює candidate beliefs і готує сесію до очищення або скорочення.

Sleep має бути органічною компресією. Його результат має зберігати сенс, емоцію, важливі факти і голос моменту. Для Chibigochi це частина ілюзії живої свідомості. Для інших проєктів це спосіб не втратити важливий контекст після завершення взаємодії.

## Reflection

Reflection - це процес довшого осмислення архіву.

Він дивиться не на одну сесію, а на накопичені спогади. Його задача - знаходити повторювані теми, стабільні патерни, суперечності, зміни в поведінці, довготривалі факти і висновки, які можуть стати candidate beliefs.

У Chibigochi reflection може побачити, що гравець часто обговорює втому, постійно повертається до певного творчого проєкту або регулярно турбується про героїню після складних рейдів. У Telegram-боті reflection може побачити довготривалі інтереси, робочі цілі або стабільні правила взаємодії. У робочому проєкті reflection може побачити повторювану архітектурну помилку або зміну принципу розробки.

Reflection створює кандидатів на стабільний висновок. Далі ці кандидати проходять правило підтвердження: достатня кількість джерел, достатня вага, відсутність явної суперечності, можливість ручного перегляду або інший контрольний механізм.

Так пам'ять отримує розвиток без хаотичних змін основи.

## LLM у внутрішній моделі

LLM у Memory Engine виконує мовні задачі, які важко зробити чистою логікою: стискання сесії, емоційний наратив, витяг цитат, формування тегів, попередня оцінка значущості, пошук суперечностей, формулювання candidate belief.

Rust core залишається координатором. Він визначає, яку задачу треба виконати, які дані передати LLM, який формат результату очікується, як перевірити відповідь і як застосувати результат до пам'яті.

Промпти для таких задач мають бути частиною Memory Engine як окремі версіоновані артефакти. Промпт sleep-компресії, промпт scoring, промпт reflection і промпт fact-checking мають розвиватися разом із ядром, тестами і історією змін.

Для Chibigochi особливо важливо, щоб поведінка героїні народжувалася з Core, Session, Archive і поточного стану. Стратегія пам'яті має підтримувати memory-driven context: особистість проявляється через накопичений досвід, стан і ситуацію, а не через випадкові готові репліки.

## Надійність даних

Memory Engine має ставитися до пам'яті як до довготривалих даних.

Потрібні надійне збереження, резервне відновлення, manifest із версіями схем, перевірка цілісності і міграції. Це важливо для мобільної гри, де застосунок може завершитися в незручний момент. Це так само важливо для ботів і робочих проєктів, де пам'ять може накопичуватися місяцями.

У межах стратегії це означає: пам'ять є цінним станом продукту. Її збереження, відновлення і розвиток мають бути частиною ядра, а не випадковим додатком інтеграції.

## Memory Engine як окремий проєкт

Memory Engine має розвиватися як самостійний продукт.

Це означає, що в нього має бути власний репозиторій, власна структура, власні версії, власні тести, власна документація, власні релізи, власна історія рішень і власна політика сумісності. Chibigochi, Telegram-бот та інші проєкти мають підключати його як залежність або модуль, а не копіювати його логіку всередину себе.

Окремий репозиторій фіксує правильну відповідальність. Memory Engine стає спільною технологією для кількох продуктів. Усі важливі зміни логіки пам'яті, контрактів, тестів і адаптерів проходять через цей репозиторій.

Для Chibigochi це означає спокійнішу розробку гри. Гра може розвивати UI, баланс, рейди, меблі, PC-магазин, сцени, анімації і AgentGateway. Memory Engine паралельно розвиває довготривалу пам'ять, sleep, recall, Archive, Core, міграції і перевірки.

Для Telegram-бота це означає спільну пам'ять з тією самою філософією, але з іншим адаптером. Бот користується ядром через Python-шар, передає повідомлення і отримує контекст без власної паралельної реалізації всієї системи.

Для майбутніх проєктів це означає нижчий поріг входу. Новий продукт має реалізувати адаптер і свої доменні події, а не створювати довготривалу пам'ять із нуля.

## Документи окремого проєкту

У Memory Engine має бути кілька типів документації.

Стратегічний документ пояснює намір, форму продукту і принципи. Архітектурний документ описує компоненти ядра та адаптерів. Контракти даних описують формати подій, спогадів, Core-фактів, recall-запитів, manifest і міграцій. Інтеграційні документи описують підключення до Godot, Python/Telegram та майбутніх середовищ. Тестові сценарії описують очікувану поведінку пам'яті на рівні прикладів.

Ці документи мають виростати з цієї стратегії поступово. Стратегія задає напрям. Детальні ТЗ мають уточнювати окремі частини без зміни основного наміру.

## HISTORY.md

Memory Engine має мати власний `HISTORY.md`.

Цей документ потрібен для фіксації важливих змін, корекцій, відкликаних тверджень, зміни контрактів, міграцій, проблем безпеки, змін у тестах, змін у benchmark-claims і рішень, які впливають на інтеграції.

Приклад MemPalace важливий не як стиль оформлення, а як культура відповідальності. У ньому історія використовується для чесного запису того, що було заявлено, що виявилося неточним, що виправлено, які метрики змінилися і які публічні твердження більше не підтримуються.

Для Memory Engine така історія критична, бо пам'ять є чутливим шаром. Якщо змінюється алгоритм recall, інтеграції мають знати, що результати пошуку спогадів можуть стати іншими. Якщо змінюється sleep-компресія, якість архівних спогадів може змінитися. Якщо змінюється формат Archive, старі дані потребують міграції. Якщо змінюється правило промоції в Core, поведінка агента може стати іншою на довгій дистанції.

`HISTORY.md` має фіксувати події такого типу:

- breaking changes у контрактах;
- зміни форматів Core, Session, Archive, міграцій, recall, scoring, decay, sleep або reflection;
- виправлення помилок, які могли пошкодити пам'ять або змінити поведінку інтеграцій;
- відкликання неточних тверджень про якість або можливості;
- зміни у benchmark-методології;
- проблеми безпеки;
- зміни, які потребують дій від інтеграцій.

Це має бути документ довіри. Він показує не лише що додано, а й що було виправлено, уточнено або переглянуто.

## DEVLOG.md

Поруч із `HISTORY.md` має бути `DEVLOG.md`.

`DEVLOG.md` фіксує робочий процес: що планувалося, що зроблено, які задачі рухалися, які рішення обговорювалися, які частини ще відкриті. Це щоденник розробки.

`HISTORY.md` фіксує наслідки для продукту: важливі зміни, корекції, релізні рішення, міграції, зміни контрактів і речі, які мають знати користувачі Memory Engine.

Для цього проєкту корисно мати обидва документи. DEVLOG допомагає розробляти. HISTORY допомагає довіряти.

## Версії і сумісність

Memory Engine має мати версії.

Версія ядра показує, яка логіка пам'яті використовується. Версія схем показує, у якому форматі збережені Core, Session, Archive і manifest. Версія адаптера показує, з якою версією ядра він сумісний.

Це особливо важливо для Chibigochi. Гравець може мати save, створений старою версією гри. Нова версія Memory Engine має розпізнати формат, виконати міграцію, оновити manifest і повідомити інтеграцію про результат.

Для Telegram-бота це так само важливо. Бот може роками накопичувати пам'ять. Оновлення ядра має враховувати існуючі дані, а не вимагати ручного переписування архіву.

Сумісність має бути описана явно:

- які версії схем підтримуються;
- які версії потребують міграції;
- які зміни є breaking;
- які адаптери сумісні з якими версіями ядра;
- які дані можна відновити автоматично;
- які ситуації потребують ручного рішення.

Це робить Memory Engine придатним для довгого життя.

## Тести

Memory Engine має тестуватися окремо від хост-проєктів.

Тести мають покривати поведінку Rust core, сценарії пам'яті, міграції, recall, sleep, reflection і роботу адаптерів. Для LLM-залежних задач тести мають перевіряти структуру результату, збереження ключового сенсу і відповідність очікуваному контракту.

Ці тести мають захищати основну цінність проєкту: пам'ять можна оновлювати без втрати довіри.

## LLM adapters і промпти як частина продукту

Memory Engine має мати окремий шар для LLM adapters.

LLM adapter відповідає за виконання мовних задач: compression, scoring assist, reflection assist, fact-checking assist, quote extraction, tag proposal. Провайдер може змінюватися, але Memory Engine має зберігати однакову форму задачі і перевіряти результат.

Промпти мають версіонуватися разом із проєктом. Зміна промпта sleep-компресії може змінити якість Archive. Зміна промпта reflection може змінити candidate beliefs. Зміна промпта scoring може змінити відбір спогадів. Такі зміни мають проходити через HISTORY, тести і версію.

Це робить LLM частиною контрольованої архітектури, а не випадковим зовнішнім текстом.

## Як має виглядати успіх

Успіх Memory Engine можна описати через практичний сценарій.

Chibigochi підключає Godot-адаптер до Rust core. Гра передає події, стани, діалоги, сон, рейди, покупки і lifecycle-тригери. Memory Engine веде Session, стискає день у Archive, повертає релевантні спогади для AgentGateway, формує candidate beliefs і зберігає дані надійно.

Telegram-бот підключає Python-адаптер до того самого Rust core. Бот передає повідомлення, завершення сесій, команди і фонові тригери. Memory Engine повертає довготривалий контекст, recall, архівні спогади і результати reflection.

Третій проєкт підключає власний адаптер. Він отримує ту саму тришарову пам'ять, ті самі принципи ваг, той самий sleep, той самий HISTORY-підхід і ту саму політику версій.

Усі ці проєкти залишаються різними продуктами. Спільною стає пам'ять.

## Підсумок стратегії

Memory Engine має бути окремим Rust-проєктом для повторного використання тришарової пам'яті.

Його головна мотивація - прибрати дублювання пам'яті між Chibigochi, Telegram-ботом та іншими проєктами і створити одне місце розвитку для Core, Session, Archive, sleep, recall, reflection, ваг, міграцій і надійного збереження.

Rust core є центральним джерелом логіки. Адаптери підключають це ядро до конкретних середовищ. Хост-проєкти передають події і стани. Memory Engine повертає контекст, спогади, висновки і результати обробки.

Проєкт має розвиватися з власною документацією, HISTORY, DEVLOG, тестами, версіями, схемами і політикою сумісності.

Цей документ є стратегічною основою. На його базі треба створювати конкретні ТЗ: архітектуру Rust core, схеми даних, план Godot-адаптера, план Python-адаптера, правила LLM adapters, формат HISTORY, сценарії тестування і порядок першого MVP.

## Update 2026-05-17: керованість для людини

До стратегії додається окремий принцип: Memory Engine має бути зручним не тільки для коду, а й для людини, яка володіє проєктом, перевіряє пам'ять, змінює промпти, тестує провайдерів і хоче розуміти, що саме відбувається.

Цей принцип не означає, що потрібно наперед вигадувати всі майбутні промпти, схеми або папки. Навпаки: структура має задавати місця і правила. Нові частини з'являються тільки тоді, коли вони реально потрібні, але одразу кладуться в правильне місце і отримують людське пояснення.

### Провайдери, моделі і ключі

Rust core не має фіксувати конкретних LLM-провайдерів, моделей або API-ключів. Ядро не повинно знати, що зараз використовується OpenAI, Google, Anthropic, Kimi, DeepSeek або інший провайдер.

У ядрі мають існувати задачі та ролі моделей, наприклад:

- потужна модель для складного мислення;
- середня модель для звичайних задач;
- швидка дешева модель для простих агентів або допоміжних операцій.

Конкретний провайдер, конкретні назви моделей і тестові ключі мають обиратися за межами ядра: у конфігурації хост-проєкту, адаптера або локального середовища розробки. Memory Engine може просити виконати задачу певного типу, але вибір `provider + model + key` має залишатися налаштуванням, а не зашитою логікою.

У проєкті має бути зрозуміле місце для такої конфігурації. Приклад конфігурації може лежати в `config/`, але реальні локальні ключі не мають бути частиною бібліотечного коду.

### Промпти

У коді Memory Engine не має бути великих промптів. Промпти є текстовими артефактами продукту і мають жити окремо від логіки.

На старті не потрібно створювати промпти наперед. Має існувати папка `prompts/` і `README.md`, який фіксує правило:

- якщо промпт реально використовується, він лежить у `prompts/`;
- якщо промпта ще немає в реальній функції, файл під нього не вигадується;
- кожен промпт має бути описаний у README людською мовою;
- опис має пояснювати, що промпт робить, коли запускається, що отримує, що повертає, що можна змінювати і що не можна ламати без зміни контракту;
- назви класів і внутрішніх функцій не замінюють пояснення для людини.

Це потрібно для того, щоб власник проєкту міг відкрити папку промптів, зрозуміти призначення кожного тексту і змінити його без пошуку по коду.

### Пам'ять і історія сесій

Пам'ять не має зберігатися всередині коду і не має звалюватися в один великий файл, у якому після кількох сесій неможливо розібратися.

Runtime-пам'ять має мати окреме місце, наприклад `memory/`, із підпапками для шарів:

- `memory/sessions/` для окремих сесій;
- `memory/archive/` для довготривалих спогадів;
- `memory/core/` для стабільної основи пам'яті;
- `memory/manifest` або інший службовий manifest для версій, схем і стану сховища.

Кожна сесія має отримувати власну папку, а не додаватися в одну безкінечну купу. Усередині сесії можуть бути машинні файли на кшталт JSON/JSONL і читабельні Markdown-файли для людини. Машинний формат потрібен для стабільності коду, а Markdown потрібен для перевірки, розуміння і ручного аудиту.

Кожна папка пам'яті має мати коротке людське пояснення: що тут лежить, хто це створює, хто це читає, що можна редагувати руками, а що краще не змінювати без міграції або спеціального інструмента.

### Стиль майбутнього коду

Майбутня реалізація має бути розбита на зрозумілі модулі та типи. У Rust це означає не обов'язково "класи" у Python-сенсі, а чіткі `struct`, `trait`, `impl` і сервіси з окремою відповідальністю.

Конфігураційні значення, шляхи, ліміти, ролі моделей, провайдери, правила збереження і режим роботи не мають бути розкидані по коду. Вони мають бути винесені в очевидні конфігураційні структури та файли.

Цей update є частиною стратегічної межі проєкту: Memory Engine має бути не тільки технічно правильним, а й прозорим, редагованим і перевірним для людини.

## Update 2026-05-18: спадкоємність із індустрії

У `wiki/pages/research/` зібрані огляди agentic-архітектур, мультимодальних стеків, провайдерів і зразок HISTORY-документа з MemPalace. Цей розділ фіксує, що саме з них Memory Engine свідомо приймає, а що навмисно не приймає. Без вичерпного академічного огляду — лише робочі рішення для нашого скелета.

### MemPalace як основний референс

MemPalace - найближча відома open-source memory-система до нашої місії. Ми не копіюємо її код і не намагаємось її випередити на бенчмарках до того, як ядро стабілізується, але приймаємо її дисципліну довіри:

- `HISTORY.md` - документ публічних корекцій, ретракцій і поведінкових змін, а не changelog у стилі "added X, fixed Y";
- newest first, кожен запис - окремий датований блок;
- структура запису: контекст -> що змінено -> що відкликано -> що залишається істинним -> що робимо далі -> подяки;
- будь-яке benchmark- або performance-твердження супроводжується reproducibility-anchor: tag, dataset, seed, шлях до result-файлів у репо;
- захист від over-correction окремою секцією "що залишається істинним", щоб ретракція не знищила реальні робочі речі.

Формат HISTORY-template вже зафіксовано у шапці нашого `HISTORY.md` за цим зразком.

### Що з agentic-research приймаємо як структурні заборони

Чотири анти-паттерни з огляду 2026 у нас виключені за конструкцією ядра, а не за домовленістю:

1. **"Архітектура прив'язана до одного провайдера"** - ядро системи фізично не виконує мережевих викликів і не знає провайдерів, моделей або API-ключів. Це гарантує `EngineConfig`, який містить лише `memory_dir` і `limits`.
2. **"Один гігантський промпт"** - промпти живуть окремо у `prompts/<prompt_id>.md`, мають свою версію і структуру, не вклеюються в код.
3. **"Прихований пошук"** - `RecallResult` несе `relevance_score`, `relevance_explanation` і `RecallDebug`. Хост точно знає, чому конкретний спогад потрапив у видачу.
4. **"Infinite loops / denial of wallet"** - ядро не виконує LLM, не може зациклитись. `PendingTask` створюється один раз, `resume()` валідує schema, хост контролює retries.

Ці заборони - не косметика, а структурна частина стратегії. Будь-яке майбутнє рішення, що їх послаблює, проходить через `HISTORY.md`.

### Що з agentic-research беремо як майбутню технічну позицію

**Observation masking для стиснення довгого хвоста сесії** (JetBrains Research, грудень 2025). Коли реалізовуватимемо session-tail compression (наразі це відомий борг), стандартом обираємо observation masking - заміну старих подій компактним маркером типу `[N events compressed]` - перед LLM-summary. За дослідженням це дає 50%+ cost reduction і часто кращу якість, ніж LLM-summary. LLM-summary залишається для archive entry, не для живого хвоста сесії.

### Що з provider-landscape приймаємо у мапінгу ролей

Наш `ModelRole` (`Reasoning` / `Balanced` / `Fast`) узгоджується з типовою тришаровою cascade-системою з огляду провайдерів. Конкретний мапінг живе **не в ядрі**, а у хост-адаптері. Типове напрацювання для Smartest-бота, яке ляже в `HostLlmConfig`:

- `Reasoning` - клас Claude Sonnet 4.6 / Opus 4.6 / o3;
- `Balanced` - клас Gemini 2.5 Flash / GPT-4.1 Mini;
- `Fast` - клас GPT-4.1 Nano / Gemini Flash-Lite / DeepSeek V3.2.

Це орієнтир, не зашитий контракт. Хост вільний обирати інші моделі у тих самих tier'ах.

### Що з provider-landscape беремо як майбутній вибір для Stage 2/3 recall

Коли вмикатимемо Stage 2 (embeddings re-rank) і Stage 3 (LLM rerank) у recall-пайплайні, перший кандидат:

- **Stage 2 embeddings** - Cohere Embed v3 (`$0.10/1M`), або OpenAI text-embedding-3-small, або Gemini embeddings free tier;
- **Stage 3 LLM rerank** - Cohere Rerank 3.5 як спеціалізована модель, або стандартна LLM через `PendingTask` типу `recall_rerank`.

Конкретний вибір живе в хост-конфігу. Ядро лише фіксує `embedding_model_id` у архівних записах для майбутньої міграції.

### Чого ми навмисно не беремо

- Повних agent-фреймворків (OpenAI Agents SDK, LangGraph, CrewAI, AutoGen). Наш scope інший: ми memory layer, а не orchestrator. Custom-orchestrator на стороні хоста простіший і дешевший за наш use case.
- MCP як основний контракт інтеграції на v0.1. Наш `PendingTask` природніший для embed-of-memory сценарію. MCP-facade залишається відкритою опцією для v0.3+, якщо з'явиться зовнішній агент (Claude Code, OpenAI Agents SDK), що захоче користуватись пам'яттю як tool'ом.
- Conversation-only моделей пам'яті (OpenAI Sessions з SQLite/Redis, LangGraph checkpointing). Наша Session - структуровані події з ваг, importance hints, тегами і темою, а не plain messages.

### Оцінка проти індустрії - не зараз

Бенчмарки на кшталт LongMemEval і LoCoMo згадані у джерелах. Memory Engine **не** робить публічних claims про recall accuracy, R@5, R@10 або інші числа до того, як ядро стабілізується і пройде reproducibility-перевірку на конкретному tag'у з committed result-файлами. Будь-яке таке твердження після стабілізації проходить через `HISTORY.md` із reproducibility-anchor.

## Update 2026-05-23: напрям v0.2 — memory units, validation, reflection, privacy

Після live-тестів v0.1 наступний стратегічний крок — не embeddings і не нова інженерна обв'язка, а якісна форма "живої пам'яті": окремі memory units, fidelity validation, reflection candidates і контрольоване забування.

### Adaptive Stable Core

Стратегічна ціль v0.2 — **живе Ядро**, яке валідує знання, адаптується під час і після розмови, але не втрачає стабільність. Це не статичний файл фактів і не хаотичний потік нових висновків. Це повільний шар пам'яті, який змінюється тільки тоді, коли досвід пройшов перевірений шлях.

Ядро адаптується через:

- validated memory units, зроблені зі стиснених сесій;
- reflection candidates, які знаходять повторювані або сильні патерни;
- recall feedback, коли спогад стає важливішим через повторне використання;
- contradiction/contested flow, коли новий досвід ставить під сумнів старий Core-факт;
- forgetting/review, коли рутина виходить із активного recall, але audit trail лишається.

Ядро залишається стабільним через:

- fidelity validation перед тим, як стислий спогад стане нормальним матеріалом для Archive або Core;
- заборону агентам напряму писати істину в Core Store;
- lifecycle `candidate` -> `ready_for_review` -> `confirmed` / `rejected` -> `active CoreFact`;
- повільну зміну Core-фактів через `contested` і `deprecated`, а не миттєве перезаписування;
- людино-читабельний audit trail для кожної зміни, яка впливає на довготривалу пам'ять.

Тобто Memory Engine має бути адаптивним, але не легковірним. Він має вчитись із розмови, але не підміняти стабільну основу випадковою відповіддю моделі.

### Агенти не правлять істину напряму

LLM-агенти у Memory Engine не мають напряму записувати істину в Core. Вони виконують ролі:

- виділяють атомарні memory units;
- стискають raw events у короткі тези;
- перевіряють, чи стиснення не викривило джерело;
- пропонують шар для спогаду: archive, core_candidate, routine або needs_revision;
- формують candidate beliefs;
- знаходять можливі суперечності.

Core Store змінюється тільки через керований lifecycle і підтвердження. Це зберігає головний принцип: модель може думати широко, але структура пам'яті не дозволяє їй самовільно переписати стабільну основу.

### Memory unit як атомарний спогад

Спогад має бути відокремленою одиницею, а не тільки частиною великого archive JSON. Один memory unit — це один змістовний епізод або висновок:

- коротка теза у формі `подія -> висновок`;
- джерела, з яких він зроблений;
- вага/важливість;
- fidelity status;
- статус шару: active archive, core candidate, forgotten, rejected, needs revision.

У storage memory unit може мати повний технічний ID. У prompt і agent-review треба використовувати короткі локальні мітки (`m1`, `m2`, `m3`), щоб не витрачати токени на довгі числові ідентифікатори. Це також готує систему до майбутнього vector storage: embed'ити треба атомарні units, а не великий змішаний archive entry.

### Fidelity validator

Стиснення не має романтизувати, перебільшувати або додумувати. Тому v0.2 має отримати fidelity validator: агент, який порівнює memory unit із raw/source events і відповідає, чи unit:

- `valid`;
- `too_broad`;
- `unsupported`;
- `distorted`;
- `missing_key_detail`;
- `needs_revision`.

Validator не вирішує, що важливо для Core. Він перевіряє тільки чесність трансформації: чи відповідає compressed memory тому, що реально було у джерелі.

Для дорогих перевірок і формування Core-тез не треба давати великій моделі всю розмову. Memory unit має посилатись на конкретні source events, а система має збирати малий evidence pack: тільки ті репліки й локальний контекст, без яких неможливо чесно перевірити або сформулювати тезу. Дорога модель використовується там, де обсяг малий, а ризик високий: кандидат у Core, суперечність із Core, важливий емоційний спогад. Її задача — не переказати діалог, а сформулювати коротку, щільну і перевірену тезу.

Prompt для відповіді користувачу має зберігати геометрію шарів. Core, довга памʼять, коротка памʼять, поточний стан і поточна репліка користувача мають бути видимими окремими секціями або тегами. Це не стилістика, а частина памʼяті: модель не повинна плутати давній спогад із живим діалогом або починати відповідь так, ніби transcript щойно почався.

Маршрутизація моделей є частиною економіки якості. Масові задачі можуть виконуватись balanced-моделлю. Критичні перевірки на малому evidence pack і фінальне формулювання Core-тези мають отримувати reasoning-модель через `role_hint`, бо там ціна помилки вища, а обсяг малий.

### Reflection як перший справжній v0.2-крок

Reflection має працювати по validated memory units, compact memories і archive tracks. Її задача — знайти повторювані або сильні патерни, яких ще немає в Core, і створити candidate beliefs із evidence. На першому етапі кандидати підтверджуються вручну через host-команди. Auto-confirm можливий тільки після того, як на реальних даних буде видно якість candidates.

### Забування як друга половина живої пам'яті

Archive не має перетворитися на склад. Забування реалізується не фізичним видаленням, а статусом або переміщенням у `forgotten/` з audit trail. Forget-review agent має отримувати старі units із metadata (freshness, weight, recall_count, emotional markers, core links) і рекомендувати, що природно прибрати з default recall. Важливі, емоційні або Core-linked спогади мають бути захищені від автоматичного забування.

### Vector storage — пізніше і тільки opt-in

Embeddings і vector storage не потрібні, доки Core/reflection/fidelity/forgetting не стали стабільними. Частина користувачів не захоче зберігати настільки детальну семантичну репрезентацію себе. Тому vector storage має бути явною опцією: "з embeddings" або "без embeddings". Без embeddings Memory Engine має лишатися повністю працездатним через структурний recall, compact memory і reflection по validated units.
