# DEVLOG

## Правила

`DEVLOG.md` ведеться українською мовою.

Кожен робочий запис має бути зрозумілим людині, яка повернеться до проєкту пізніше і захоче швидко згадати, що відбувалося, чому саме так робили і які питання залишилися відкритими.

Бажаний формат запису:

- **Час** - дата і час запису, якщо є доступ до годинника.
- **Проблематика** - яка проблема або потреба була перед роботою.
- **Задум** - яке рішення або напрям обрали перед змінами.
- **Що робили** - які файли, структури або рішення змінювали.
- **Що зроблено** - детальний результат, щоб не треба було відновлювати його по коду.
- **Проблеми чи виклики** - що було незрозуміло, ризиковано, спірно або потребує майбутньої уваги.
- **Фідбек користувача** - важливі правки, зауваження або принципи, які дав власник проєкту.
- **Наступні кроки** - що логічно робити далі, якщо це вже зрозуміло.

`DEVLOG.md` фіксує робочий процес і контекст розробки. Важливі продуктові зміни, breaking changes, міграції, зміни контрактів, зміни поведінки пам'яті або речі, які мають знати інтеграції, потрібно також фіксувати в `HISTORY.md`.

Створено приклад конфігурації `config/llm.example.toml`. Він показує форму для активного провайдера, шляхів, provider-блоків і ролей моделей: потужна reasoning-модель, середня balanced-модель і швидка fast-модель. Реальні ключі мають бути в локальних ignored-файлах, а не в бібліотечному коді.

У `prompts/README.md` зафіксовано правило: промпти додаються тільки тоді, коли реальна функція їх використовує. Порожні майбутні промпти не створюються. Кожен реальний промпт має бути описаний людською мовою: що робить, коли запускається, що отримує, що повертає, що можна змінювати і що не можна ламати без зміни контракту або тестів.

У `memory/` створено базову структуру для майбутнього runtime-сховища. Пам'ять має бути поза кодом і розбита на `sessions/`, `archive/` і `core/`. Для сесій зафіксовано правило: кожна сесія отримує власну папку, а не додається в один великий файл.

У `HISTORY.md` зафіксовано призначення файлу: він не для щоденних нотаток, а для важливих продуктових змін, сумісності, схем, міграцій, поведінки пам'яті і довіри до проєкту.

Поки немає реалізації Rust core, тому створена структура є підготовкою до роботи, а не робочим engine.

**Фідбек користувача:**
Власник проєкту уточнив, що не треба вигадувати промпти до початку реальної реалізації. Потрібно фіксувати логіку і правила, а не створювати файли на майбутнє.

Після створення першого DEVLOG власник проєкту уточнив, що DEVLOG має вестися українською і містити правила заповнення: проблематика, задум, що робили, що зроблено детально, проблеми чи виклики, фідбек користувача і час запису, якщо доступний годинник.

### Запис 2

**Час:** 2026-05-17 (вечір того ж дня)

**Проблематика:**
Стратегія була зафіксована, але кожне практичне рішення лишалось відкритим: як ядро викликає LLM, як влаштований recall, у чому зберігати дані, з якого хоста починати. Без цих рішень писати Rust-код було рано — будь-який перший trait одразу б накинув форму, яку довелось би переробляти.

Також у словнику стратегії було двозначне використання слова "core": і як назва бібліотеки, і як назва шару пам'яті. Це треба було розвести явно, інакше подальші документи і код плутатимуть.

**Задум:**
Зробити чотири несучі архітектурні вибори свідомо й одразу, оптимізуючи "як правильно для бібліотеки, що має жити роками", а не "як швидше зробити MVP". Потім зафіксувати їх у самодостатньому документі `docs/architecture.md`, який зможе прочитати людина або ШІ, що знає лише стратегію, і зрозуміти задум без зовнішнього контексту.

**Що робили:**
Розвели терміни: `ядро системи` = вся Rust-бібліотека; `шар Ядро` = один із трьох шарів пам'яті.

Прийняли чотири архітектурні рішення:

- **А. Як ядро системи спілкується з LLM:** задачно-орієнтована модель. Ядро не виконує мережевих викликів; повертає `PendingTask`-об'єкти, які хост виконує через свого LLM-провайдера і повертає результат через `resume()`. `PendingTask` серіалізується на диск для виживання при lifecycle-обриві.
- **Б. Recall:** трирівневий гібридний пайплайн, закладений від початку. Стадія 1 (фільтр + детермінована формула) активна на v0.1. Стадія 2 (embeddings re-rank) і стадія 3 (LLM re-rank) мають готову форму, вмикаються конфігом у v0.2+. Архівні спогади з v0.1 уже мають поле `embedding_model_id` для майбутньої сумісності.
- **В. Сховище:** файли як джерело правди. JSON/JSONL для машинно-читабельних шарів, Markdown для людських версій. Доступ через trait `Storage` — індексована імплементація може з'явитись пізніше без зміни формату даних. Crash safety через write-temp-rename + журнал мульти-файлових операцій.
- **Г. Перший хост:** Telegram-бот через PyO3. Godot-адаптер через GDExtension — другим у v0.2+. Контракти у документі валідовані прикладами з обох хостів (бот і Chibigochi), щоб уникнути архітектури, заточеної під один домен.

Створено `docs/architecture.md` v0.1. Документ покриває: термінологію, чотири основні потоки даних, обґрунтування кожного з чотирьох рішень, контракт події (з прикладами з обох хостів), три шари пам'яті у деталях, recall pipeline, PendingTask механіку, sleep як двостадійний процес, reflection (відкладено на v0.2+), storage layout, версіонування і сумісність, конкурентний доступ, адаптери, чіткий скоуп v0.1 vs v0.2+, і список того, що документ навмисно не задає.

**Що зроблено:**
Архітектурний документ написаний так, щоб бути самодостатнім — читач зі стратегією на руках і без жодного знайомства з цією розмовою має могти зрозуміти задум.

Зафіксовані обов'язкові поля події (type, source, timestamp, session_id, payload, schema_version) і опціональні (tags, theme, emotional_tone, links, importance_hint, processing_mode). Кожне поле супроводжене прикладами.

Описані три шари: Сесія (events.jsonl + session.md, з лімітами і попереднім стисканням живого хвоста), Архів (структуровані спогади з вагою, freshness, тегами, темою, наративом, цитатами, embedding_model_id), шар Ядро з розрізненням Core Store (стабільна основа у файлах) і Core Context Package (готовий пакет для LLM, що збирається з Core Store плюс поточний доменний стан від хоста).

Зафіксовані правила промоції в шар Ядро на v0.1 (≥3 джерела з різних сесій, поріг сумарної ваги, відсутність суперечності в останніх K записах, обов'язковий ручний огляд власника). Автоматична промоція без огляду — задача v0.2+.

Описаний sleep як двостадійний процес: алгоритмічна частина (топ-N, групування, базовий спогад) виконується синхронно і зберігає попередній архівний запис, навіть якщо LLM-доробка ніколи не відбудеться. LLM-доробка через `PendingTask` оновлює той самий запис, додаючи наратив, цитати, скориговану вагу.

Storage trait виділений як абстракція, дефолтна імплементація — файлова. Розкладка: `memory/sessions/<id>/`, `memory/archive/<YYYY>/<MM>/`, `memory/core/store/<category>.*`, `memory/core/candidates/`, `memory/tasks/`, `memory/journal/`.

Manifest (`memory/manifest.json`) тримає `engine_version`, `schema_versions` по шарах, `active_embedding_model_id`, `last_migration_at`.

Промпти живуть у `prompts/<prompt_id>.md` із YAML-frontmatter'ом (prompt_id, version, description, inputs, output_schema, role_hint). На v0.1 буде один реальний промпт — `sleep_compression.md`, що з'явиться разом із кодом sleep-stage-2.

Чіткий розподіл v0.1 vs v0.2+. У v0.1: ядро + Storage trait + FileStorage + три шари + recall стадія 1 + sleep двостадійний + PendingTask + manifest + crash safety + один RwLock + Python-адаптер + інтеграція з ботом + один реальний промпт + базові тести. У v0.2+: recall стадії 2/3, reflection, автоматична промоція, ротація сесій, Godot-адаптер, fine-grained locking, індексоване сховище, розширений набір PendingTask-типів.

**Проблеми чи виклики:**
Перша відповідь ШІ на стратегію була подана у консультантському тоні — список ризиків і питань без явної позиції. Власник попросив напарника, а не критика. Зафіксували у пам'яті ШІ як принцип спілкування для цього проєкту: брати позицію явно, конвертувати ризики у конкретні рішення з обґрунтуванням, говорити "ми" замість "проєкту слід".

Також ШІ плутав терміни "ядро" і "Core" — обидва зі стратегії, але в різних сенсах. Власник зупинив і запропонував розвести: `ядро системи` для бібліотеки, `шар Ядро` для шару. Усі подальші документи і код мають триматись цих імен.

Питання, що залишаються відкритими і чекають на наступний документ:

- Точні JSON-схеми подій, архівних спогадів, Core-фактів, PendingTask-результатів.
- Точні формули scoring, decay, recall ranking (базова форма є, конкретні коефіцієнти треба підбирати емпірично).
- Структура Rust-крейту (модулі, trait'и, типи) — пишеться разом із початком коду.

**Фідбек користувача:**
Власник чітко сформулював: рішення приймати з оптимізацією "як правильно для довгої дистанції", не "як швидше зробити перший прототип".

Власник попросив написати архітектурний документ так, щоб він був самодостатнім — ШІ читач без контексту розмови, але зі стратегією, мав могти зрозуміти задум. Це позначило, що деталі важливі і їх не можна стискати до bullet-list'у, де губиться обґрунтування.

**Наступні кроки:**
Після затвердження `docs/architecture.md` v0.1 власником — переходити до контрактів даних (точні JSON-схеми) і до першого Rust-коду: `Cargo.toml`, скелет крейту з модулями, перший trait `LlmAdapter`, перший тест із mock-resume. Промпт `sleep_compression.md` з'являється разом із кодом sleep-stage-2.

Але після довгої серії про імʼя "Аврора" і персональні факти на питання "А про літаки?" відповів, що про літаки розмови не було. Причина: recent context був обмежений 16 подіями, тобто приблизно 8 turn-ами, і тема літаків була витіснена з prompt.

Окремо runtime log підтвердив, що попереднє "зависання" не було crash-ем: bot отримав старі pending updates після рестарту і нормально їх обробив.

**Задум:**
Зробити два різні короткострокові контексти:

- `session topic trace` - ширший trace тем поточної сесії для питань "про що ми говорили?";
- `recent session context` - останні репліки для локальних follow-up посилань типу "їх", "це", "а як щодо".

Також треба зберігати Telegram polling offset у runtime, щоб після рестарту bot не відповідав повторно на вже оброблені updates.

**Що робили:**

- оновлено `hosts/telegram_gemini_bot/bot.py`;
- оновлено `hosts/telegram_gemini_bot/README.md`;
- оновлено `docs/local-development.md`.

**Що зроблено:**

- `RECENT_CONTEXT_LIMIT` збільшено з 16 до 40 подій;
- додано `SESSION_TRACE_EVENT_LIMIT = 120`;
- додано `SESSION_TRACE_CHAR_LIMIT = 12000`;
- prompt тепер містить:
  - `Session topic trace`;
  - `Recent session context`;
  - `Archive memory context`;
  - current user message;
- system instruction явно каже використовувати `session topic trace` для питань про те, що обговорювали в поточному чаті;
- Telegram offset зберігається в:

```text
hosts/telegram_gemini_bot/runtime/state/telegram_offset.json
```

Offset оновлюється після кожного обробленого update.

**Проблеми чи виклики:**
Це не замінює справжню довгострокову памʼять. Для довгих діалогів усе одно треба запускати `/sleep`, щоб теми переходили в archive memory. Але для live-чату тепер не має бути ситуації, де тема випала після 8 turn-ів.

**Фідбек користувача:**
Користувач показав transcript, де bot памʼятав імʼя "Аврора", імʼя користувача і вік, але не згадав попередню тему про літаки.

**Перевірки:**

- `python -m py_compile hosts/telegram_gemini_bot/bot.py hosts/telegram_gemini_bot/launch_gui.py` проходить;
- `git diff --check` проходить.

**Наступні кроки:**
Перезапустити bot і повторити тест на довшу сесію: літаки -> риболовля -> імʼя -> персональні факти -> питання "А про літаки?".
